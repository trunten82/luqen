# Roadmap: Luqen v2.8.0 — Admin UX & Compliance Precision

## Overview

This milestone gives admins direct control over cross-service configuration, brand defaults, and regulation-level compliance scoping through the dashboard — eliminating config-file edits and all-or-nothing jurisdiction scans. The 22 requirements cluster into three independent feature areas (service connections UI, regulation filter, system brand guideline), each delivered as a self-contained phase that ships DB, API, and UI together.

Phase numbering continues from v2.7.0 (which ended at Phase 05), starting at Phase 06.

## Phases

**Phase Numbering:**
- Integer phases (6, 7, 8): Planned milestone work
- Decimal phases (6.1, 6.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [x] **Phase 07: Regulation Filter** — Scans, reports, and exports support an additive jurisdictions + regulations selection returning the inclusive union (completed 2026-04-05)

## Phase Details

### Phase 06: Service Connections UI
**Goal**: Dashboard admins can view, edit, and test compliance/branding/LLM service connections from a dedicated admin page, with credentials encrypted at rest in the dashboard DB and clients recreated at runtime without a restart
**Depends on**: Nothing (first phase of milestone — foundational infrastructure)
**Requirements**: SVC-01, SVC-02, SVC-03, SVC-04, SVC-05, SVC-06, SVC-07, SVC-08
**Success Criteria** (what must be TRUE):
  1. A dashboard admin opens `/admin/service-connections` and sees a list of compliance, branding, and LLM entries showing URL and OAuth client ID (secrets masked)
  2. The admin edits a connection (URL or credentials), saves, and the service client is immediately usable at runtime — no dashboard restart required
  3. A "Test connection" button reports auth + health status for each service inline
  4. With no DB entry present, the dashboard continues to work by falling back to `dashboard.config.json`, and the UI makes this fallback state visible
  5. Non-admin users receive 403 when attempting to view or edit the page; OAuth secrets are never returned from the API after save
**Plans**: 5 plans
- [x] 06-P01-storage-foundation-PLAN.md — Migration, encrypted repository, bootstrap-from-config helper
- [x] 06-P02-client-registry-PLAN.md — ServiceClientRegistry + server.ts refactor for runtime reload
- [x] 06-P03-admin-route-PLAN.md — /admin/service-connections API (GET list, POST update, POST test) with audit + reload
- [x] 06-P04-admin-ui-PLAN.md — Handlebars views, HTMX partials, i18n keys, sidebar nav
- [x] 06-P05-e2e-verification-PLAN.md — End-to-end flow + fallback + permission gating integration tests
**UI hint**: yes

### Phase 07: Regulation Filter
**Goal**: Users can scope scans, reports, and exports by any combination of jurisdictions and specific regulations, with results returning the inclusive deduplicated union — fully backwards compatible with jurisdictions-only callers
**Depends on**: Phase 06 (sequenced after; not a hard dependency but benefits from stable service connection plumbing)
**Requirements**: REG-01, REG-02, REG-03, REG-04, REG-05, REG-06, REG-07
**Success Criteria** (what must be TRUE):
  1. A user on the scan form selects one jurisdiction and two specific regulations (from other jurisdictions) and the resulting scan evaluates all of them
  2. The scan form regulation picker groups available regulations by jurisdiction so users can discover what's available
  3. The report detail page shows findings broken down per regulation, not only per jurisdiction
  4. CSV/PDF exports from a scoped scan include only findings matching the selected jurisdictions + regulations union
  5. A pre-existing integration calling the compliance API with `jurisdictions[]` only receives identical results to before — no behavioural change
**Plans**: 4 plans
- [x] 07-P01-compliance-engine-PLAN.md — Compliance API + checker engine: types, validation, cache key, regulationMatrix (REG-02, REG-03, REG-04)
- [x] 07-P02-dashboard-wiring-PLAN.md — Migration 039, compliance-client signature, orchestrator + scan POST + reports route wiring (REG-01, REG-03, REG-05)
- [ ] 07-P03-form-and-report-ui-PLAN.md — Scan form checkbox bug fix + report detail sub-tabs + shared partial + i18n (REG-01, REG-05, REG-07)
- [x] 07-P04-exports-and-regression-PLAN.md — CSV/PDF exports + REG-04 backwards-compat snapshot test (REG-04, REG-06)
**UI hint**: yes

### Phase 08: System Brand Guideline
**Goal**: Dashboard admins maintain a library of multiple system-wide brand guideline templates. Each org can consume a system guideline per-site in two ways: **link** (use as-is, live — edits propagate) or **clone** (import into org, frozen and customizable). Both resolve to the existing BrandGuideline matching pipeline with no parallel code path.
**Depends on**: Phase 07 (sequenced last; benefits from admin page patterns established in Phase 06)
**Requirements**: SYS-01, SYS-02, SYS-03, SYS-04, SYS-05, SYS-06
**Success Criteria** (what must be TRUE):
  1. A dashboard admin creates, edits, and deletes multiple system brand guidelines from a dedicated `/admin/system-brand-guidelines` page
  2. Org admins see a "System Library" tab on their `/admin/branding-guidelines` page listing every system guideline, with "Link to site" and "Clone into org" actions per entry — system guidelines themselves remain read-only from the org view
  3. A site with a linked system guideline scans using the live system content; editing the source system guideline as the dashboard admin propagates on the next scan
  4. Cloning a system guideline creates an independent org-owned guideline with `cloned_from_system_guideline_id` provenance; subsequent edits to the source system guideline do NOT touch the clone
  5. Non-admin users (no `admin.system`) cannot create, edit, or delete system brand guidelines; org admins can link or clone but never mutate a system guideline
  6. Orgs with no system-guideline involvement continue to work exactly as today — no migration or default-mode logic required; matching pipeline logs show the single BrandGuideline code path handling both org-owned and linked-system guidelines
**Plans**: 4 plans
- [x] 08-P01-data-foundation-PLAN.md — Migration 040 + branding repo listSystemGuidelines/cloneSystemGuideline + scope-aware resolver (SYS-06)
- [x] 08-P02-admin-route-and-page-PLAN.md — /admin/system-brand-guidelines CRUD route + list view + sidebar entry + i18n (SYS-01, SYS-04)
- [x] 08-P03-org-system-library-tab-PLAN.md — System Library tab on /admin/branding-guidelines + clone endpoint with HX-Redirect (SYS-02, SYS-03)
- [x] 08-P04-pipeline-integration-e2e-PLAN.md — End-to-end integration tests + single-code-path enforcement + retag compatibility (SYS-05)
**UI hint**: yes
