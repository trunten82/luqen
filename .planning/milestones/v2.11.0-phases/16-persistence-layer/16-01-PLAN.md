<schema_decision>
## Research gap closed in this plan: SubScoreDetail round-trip

The locked schema in `.planning/research/SUMMARY.md` lines 80-104 stores only the composite numbers (`overall`, `color_contrast`, `typography`, `components`) — it does NOT preserve `SubScoreDetail` (the per-dimension `{passes/fails}`, `{fontOk/sizeOk/lineHeightOk}`, `{matched/total}` records that Phase 15 plans 15-01..15-04 baked into the `SubScore` discriminated union).

A naive read path would have to synthesize a placeholder detail like `{dimension: 'color', passes: 0, fails: 0}` on every read — which is dishonest (the placeholder values lie about the underlying scan), breaks Phase 15's D-13 contract end-to-end, and would fail any round-trip equality test for the BrandScoreRepository in Plan 16-02.

**Decision (locked here, consumed by Plan 16-02):** Add ONE additional `subscore_details TEXT` column to the migration. Repository populates it on insert with a JSON object of shape `{ color: SubScore, typography: SubScore, components: SubScore }` when the top-level `ScoreResult.kind === 'scored'`, and stores `NULL` when `kind === 'unscorable'`. On read, the repository deserializes the column verbatim and reconstructs the typed `ScoreResult`.

This is **additive** to the research lock — it does not change any of the existing column names, types, or nullability decisions. It simply extends the schema by one column to preserve a contract that Phase 15 already established. Pitfall #3 is unaffected: the score columns remain `INTEGER NULL`.

### Schema Addition #2: CHECK constraint on `brand_scores.mode`

The locked SQL in research/SUMMARY.md does not include a CHECK constraint on the `mode` column. This plan adds `CHECK (mode IN ('embedded', 'remote'))` as an additional schema-level guard.

**Justification:** Defense in depth against an out-of-band writer that bypasses the typed repository (e.g., a future migration script, a manual SQLite REPL session, or a different storage adapter that hasn't been updated). The TypeScript literal union `'embedded' | 'remote'` is the primary contract; the CHECK constraint is the schema-level backstop. Test 9 in this plan exercises the CHECK constraint by attempting to INSERT a row with `mode = 'invalid'` and asserting the insert throws.

Like the subscore_details addition, this is **additive** to the locked schema — it does not rename, retype, or remove any column from research/SUMMARY.md, only narrows the accepted values for the existing `mode TEXT NOT NULL` column.
</schema_decision>

    const through042 = DASHBOARD_MIGRATIONS.filter((m) => m.id !== '043');

  it('rejects invalid mode values via CHECK constraint', () => {

  // Foreign key cascade
  // -------------------------------------------------------------------------

  it('cascades brand_scores rows when scan_records row is deleted', () => {
    const scanId = randomUUID();
    db.prepare(
      `INSERT INTO scan_records (id, site_url, status, standard, jurisdictions, regulations, created_by, created_at)
       VALUES (?, ?, 'completed', 'WCAG2AA', '[]', '[]', ?, ?)`,
    ).run(scanId, 'https://example.com', 'tester', new Date().toISOString());

    const brandId = randomUUID();
    db.prepare(
      `INSERT INTO brand_scores
        (id, scan_id, org_id, site_url, coverage_profile, brand_related_count, total_issues, mode, computed_at)

      10,
      'embedded',
      new Date().toISOString(),

      .get(brandId) as { n: number };

  });
