## Summary

Quick-polish milestone hardening the LLM prompt system and polishing org API key management UX. Two tightly-scoped phases, both independent of each other, both shipped live on the same day.

## Phases

### Phase 13: LLM Prompt Management

**Goal:** Org admins can safely customize LLM prompts without breaking capability engine contracts — structural sections are protected, free enrichment regions are editable, and any override can be cleanly reset to the system default.

**Requirements:** LLMPROMPT-01, LLMPROMPT-02, LLMPROMPT-03, LLMPROMPT-04, LLMPROMPT-05, LLMPROMPT-06

**Plans:**
- 13-01: Fence markers in 4 default templates + segment parser + byte-exact validator + 422 enriched response
- 13-02: Split-region editor UI + stale-override Migrate button + 3-part error toast (section + explanation + reset link)
- 13-03: Compare-with-default diff modal + rich reset-to-default modal using `diff@5`

**Success Criteria:**
1. User can click "Reset to default" on a customized prompt, see a confirmation dialog showing what will be lost, confirm, and have the org override removed so the prompt reverts to the system default ✓
2. Prompt editor displays a diff view showing exactly which lines differ from the system default before the user saves any change ✓
3. User cannot save a prompt edit that removes or alters a protected section — rejected with an error naming the protected section ✓
4. User can freely edit designated enrichment regions and save successfully without hitting any guardrail ✓

**Status:** Complete — 3/3 plans, 45 automated tests passing.

### Phase 14: API Key UX

**Goal:** Polish the org API key management UX — optional key expiry, automatic revoke on expiry, hard-delete for revoked keys, collapsible Revoked section, and visual distinction between manually revoked and expired keys.

**Requirements:** APIKEY-01, APIKEY-02, APIKEY-03, APIKEY-04, APIKEY-05, APIKEY-06

**Plans:**
- 14-01: Migration 042 (expires_at + partial index), ApiKeyRecord.expiresAt, repository extensions: storeKey(expiresAt), deleteKey(id, orgId), revokeExpiredKeys()
- 14-02: Form TTL selector with "Never" warning, POST whitelist validation, DELETE route, startup sweep + 24h setInterval + onClose cleanup, audit log mapping
- 14-03: GET split (active/revoked), template rewrite with `<details>` collapsible section + expired suffix, revoke handler row-move via `<template>`-wrapped OOB swaps, delete count OOB

**Success Criteria:**
1. Org admin can permanently delete any individually revoked key; the key is removed from the database and no longer appears in any list ✓
2. Revoked keys (including expired keys) appear in a collapsible "Revoked" section below active keys, collapsed by default, and can be expanded to view ✓
3. Key creation form offers a max TTL selector (30d, 90d, 180d, 1y, never) and displays a visible warning when "never" is selected ✓
4. Expired keys are automatically revoked on dashboard startup and daily thereafter, and appear in the Revoked section with an "Expired" label distinct from manually revoked keys ✓

**Status:** Complete — 3/3 plans, 44 automated tests passing + 9 UAT manual tests verified on live.

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| HTML comment fences `<!-- LOCKED:name -->` for protected sections | Clear to users, maps to UI, no runtime cost, one-time default rewrite | ✓ Good |
| Split-region editor (not single textarea) | User visually sees what can/can't be changed — no surprises on save | ✓ Good |
| diff@5 npm package for line diff | Lightweight, pure JS, no DOM dependency, server-side render | ✓ Good |
| Native `<details>` for collapsible Revoked section | Accessible by default, zero JS, works on mobile | ✓ Good |
| Startup + 24h setInterval for expiry sweep | Daily granularity acceptable for 30d+ TTLs, no per-request DB overhead | ✓ Good |
| Hard delete with `active=0` SQL guard | Two-step (revoke → delete) matches user mental model; defense-in-depth at DB level | ✓ Good |
| TTL whitelist `[0,30,90,180,365]` with 90d default | Industry-standard for CI tokens, server-side rejection prevents injection | ✓ Good |
| `<template>`-wrapped OOB swaps for table row moves | Required by HTML parser to survive tbody context (per feedback_htmx_oob_in_table) | ✓ Good |

## Issues Found and Resolved

During UAT on live (2026-04-10):

1. **Phase 13 Reset button never rendered** — `LLMPrompt.isCustom` (dashboard field) didn't match LLM service response field `isOverride`. Fixed in commit `9687aa6` — renamed interface field + updated all call sites and tests.
2. **Phase 14 status cell stacked badge + (Expired) on two lines** — default inline layout broke on mobile flex `<td>`. Fixed in commit `cf4a9fe` — wrapped in `.status-cell` inline-flex container.
3. **Phase 13 locked segments overflowing card + diff modal too narrow** — locked pre content had long unbreakable tokens, and modal had hardcoded `max-width: 560px`. Fixed in commit `bfe8093` — added `.modal--wide` variant + `overflow-wrap: anywhere` on locked content.

All three issues were caught during live UAT, fixed within the same session, and redeployed.

## Recovery Incident

**Wave 1 of Phase 14 spawned a worktree-isolated executor whose working tree was based on a stale snapshot from before Phase 13 was merged.** The executor's commits inadvertently reverted all phase 13 work. The merge back into master appeared to fast-forward but actually reversed 12 commits of phase 13 work.

Detected immediately when the merge log showed massive deletions. Recovery:
1. `git reset --hard 9687aa6` (last known good state with phase 13 + isOverride fix intact)
2. Surgical `git checkout c554074 -- <5 phase-14 files>` to preserve the phase 14 backend work
3. Re-committed as a single clean commit `eb3e513`
4. Ran waves 2 and 3 sequentially on the main working tree (no worktree isolation) to avoid the stale-base issue

Lesson: the `git reset --soft` safety check in the execute-phase workflow doesn't protect against stale working tree content when the worktree was created from an older base.
