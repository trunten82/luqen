# Requirements: Luqen v2.8.0 — Admin UX & Compliance Precision

**Defined:** 2026-04-05
**Core Value:** AI-powered accessibility compliance that adapts to each organization's jurisdiction, regulation, and brand context — with admins in control through the dashboard, not config files.

## v2.8.0 Requirements

### Regulation Filter

- [x] **REG-01**: User can select both jurisdictions and specific regulations in the same scan via the scan form
- [x] **REG-02**: Compliance check API accepts an optional `regulations[]` parameter alongside `jurisdictions[]`
- [x] **REG-03**: When both `jurisdictions[]` and `regulations[]` are provided, results include the inclusive union of all matching regulations (deduplicated)
- [x] **REG-04**: Existing callers passing only `jurisdictions[]` receive identical results as before (backwards compatible)
- [x] **REG-05**: Report detail page shows a per-regulation breakdown of findings
- [x] **REG-06**: Scan exports filter findings by the combined jurisdiction + regulation selection
- [x] **REG-07**: Scan form regulation picker shows available regulations grouped by jurisdiction

### Service Connections UI

- [x] **SVC-01**: Admin can view configured service connections (compliance, branding, LLM) on a dedicated admin page
- [x] **SVC-02**: Admin can edit service URL and OAuth client credentials for each service in the UI
- [x] **SVC-03**: OAuth client secrets are masked (write-only) in the UI and never re-displayed after save
- [x] **SVC-04**: Admin can test a service connection (auth + health) with a single button click
- [x] **SVC-05**: Service credentials are stored in the dashboard database encrypted at rest
- [x] **SVC-06**: Saving a connection change recreates the service client at runtime without requiring a restart
- [x] **SVC-07**: When the database has no connection entry for a service, the dashboard falls back to `dashboard.config.json`
- [x] **SVC-08**: Only users with dashboard admin permission can view or edit service connections

### System Brand Guideline

- [x] **SYS-01**: Dashboard admin can create, edit, and delete **multiple** system brand guidelines (colors, fonts, logo, brand name). System guidelines are visible to all orgs as read-only templates.
- [x] **SYS-02**: Org admins can **link** any system guideline directly to a site. Link mode is live — edits to the source system guideline propagate on the next scan.
- [x] **SYS-03**: Org admins can **clone** a system guideline into their org, producing an independent org-owned copy they can rename and customize freely. Subsequent edits to the source system guideline do not propagate. Provenance (`cloned_from_system_guideline_id`) is recorded.
- [x] **SYS-04**: Only dashboard admins (`admin.system` permission) can create, edit, or delete system brand guidelines. Org admins have read-only access to the system library; they can link or clone but never mutate.
- [x] **SYS-05**: The scan matching pipeline uses a single `BrandGuideline` code path. Linked system guidelines and cloned org guidelines resolve to an effective guideline before matching — no parallel code path.
- [x] **SYS-06**: Orgs with no system-guideline involvement continue to work exactly as today (org-owned guideline assigned via `site_branding`). System guidelines are additive — no migration required for existing data.

## Future Requirements

### Compliance

- **REG-FUT-01**: Regulation-first rework — make regulations the primary scope throughout scan, report, and export (deferred; additive filter sufficient for v2.8)

### Branding

- **SYS-FUT-01**: Multi-tenant brand hierarchy beyond system/org (tenant groups cascading down)
- **SYS-FUT-02**: Cross-org brand a11y score aggregation
- **BRD-FUT-01**: LLM discover-branding auto-links the scanned site to the newly created guideline, with a default-ON checkbox letting the user opt out. Prompt before clobbering an existing site→guideline link. Small addition, backlog memory: `project_backlog_branding_auto_link_site.md`.

### Service Config

- **SVC-FUT-01**: Per-environment service connection profiles (dev/staging/prod) switchable from the UI
- **SVC-FUT-02**: Service connection history / audit log
- **SVC-FUT-03**: LLM per-org OAuth client parity with compliance/branding. Add `organizations.llm_client_id` / `llm_client_secret` columns, `llm-client.ts` gains `createOrgClient(orgName, orgId)` mirroring `compliance-client.ts:802`, per-org LLM call routing falls back to system client when org has none set, and org admins can override default LLM credentials with their own. Enables per-org billing/quota separation and OAuth-identity audit. Backlog memory: `project_backlog_llm_per_org_oauth.md`.

### Documentation

- **DOC-FUT-01**: Main README gains a "Built on" / "Components & Credits" section linking every upstream open-source component (pa11y, Fastify, HTMX, better-sqlite3, Handlebars, Ollama, etc.) to its official GitHub repo or project website. Grouped by role (scanning / web / data / LLM / infra). Derived from package.json dependencies, not invented. Backlog memory: `project_backlog_readme_components_links.md`.

### MCP + Agent Companion (major — future milestone, likely v2.9.0 or v3.0.0)

- **MCP-FUT-01**: All Luqen services (compliance, branding, LLM, scanner, dashboard) expose MCP (Model Context Protocol) servers that mirror their existing REST/CLI capabilities as tools. Standard, language-agnostic automation surface.
- **AGENT-FUT-01**: In-dashboard AI agent companion — accessible globally (side panel / widget), can drive every feature via the MCP layer. Replaces or complements UI navigation for non-technical stakeholders.
- **AGENT-FUT-02**: Org-aware context — agent scopes every tool call to the current user's active org; cross-org data surfaces only for global admins explicitly asking system-wide questions.
- **AGENT-FUT-03**: Per-user persistent memory across sessions so the agent accumulates working knowledge of the user's projects, preferences, and recent activity.
- **AGENT-FUT-04**: Multimodal input — text AND speech (speech-to-text input, optional text-to-speech output). Web Speech API for MVP, server-side whisper/TTS for production.
- **AGENT-FUT-05**: Model routing through the existing LLM module capability engine so provider fallback, model priority chain, and per-org prompt overrides all apply.
- **AGENT-FUT-06**: RBAC + security — tool availability mirrors the user's permissions; state-changing calls require confirmation; every tool invocation written to `audit_log` with user, org, tool, args, outcome, latency.
- **AGENT-FUT-07**: Cost controls — per-user/per-org token budgets, session limits, conversation truncation strategy.
- **Backlog memory:** `project_backlog_mcp_agent_companion.md` (full detail — why, how, constraints, scope estimate).

## Out of Scope

| Feature | Reason |
|---------|--------|
| Regulation-first data model rework | Additive `regulations[]` filter is backwards compatible and sufficient |
| Custom `system-admin` permission role | Dashboard admin role is sufficient for managing system brand guideline |
| Multi-tenant brand hierarchy | System + org-level opt-in is sufficient for v2.8 |
| Service connection audit log | Non-blocking; revisit after base CRUD ships |
| Plaintext credential display in UI | Security constraint — secrets are always write-only |
| Config-file-only mode for service connections | DB-backed with config fallback is the target; pure config-file mode is legacy |
