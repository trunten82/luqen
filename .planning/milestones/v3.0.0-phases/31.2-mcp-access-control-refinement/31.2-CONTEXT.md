# Phase 31.2: MCP Access Control Refinement — Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

**What this phase delivers:**
- New `mcp.use` RBAC permission, per-org scoped, as the explicit gate for opening an MCP connection to the dashboard. System admins bypass the per-org check.
- Tool-list filtering shifts to **RBAC ∩ scope** (defense-in-depth): Phase 30.1's scope-filter rules remain, but every surviving tool must ALSO match the user's effective RBAC permissions resolved via `resolveEffectivePermissions(user, orgId)`.
- OAuth `scopes_supported` narrows to `['read', 'write']` — `admin.system`, `admin.org`, `admin.users` retired as OAuth scopes. Admin-tool visibility now comes exclusively from the user's real RBAC, not from a broad scope-bundle request.
- Org-scoped DCR client management — `/admin/clients` surfaces DCR'd OAuth clients filtered by the viewer's org (admin.system sees all; admin.org sees own-org only). Cross-org operations forbidden and audited.
- `registered_by_user_id` column added to `oauth_clients_v2`, back-filled at FIRST consent (not at DCR registration, which is pre-auth per RFC 7591).
- Service-side MCP 401 responses on compliance / branding / llm emit the same `WWW-Authenticate: Bearer resource_metadata="..."` header as the dashboard (Phase 31.1 inline fix extended to all 4 packages — required for Phase 32's per-service agent token path).
- SQL migration back-fills `mcp.use` onto every existing user × org pair to preserve day-1 continuity for the clients already live from Phase 31.1 smoke.
- Dashboard `/oauth/token` removes the `client_credentials` grant support; dashboard AS is now exclusively for user flows. Service-to-service internal bootstrap continues to use per-service `/api/v1/oauth/token` (D-10 invariant from 31.1 untouched).

**What this phase does NOT deliver:**
- Claude Desktop / external MCP client changes — they keep auto-requesting `read write` (their natural default now that admin.* is retired); dashboard serves them the narrowed user-aware surface automatically.
- New cross-service tool exposure for the agent — Phase 32 scope.
- UI redesign of `/admin/roles` beyond adding `mcp.use` — per-permission add is additive.
- Per-scope consent checkboxes — still all-or-nothing per 31.1 D-21.
- Removal of legacy `DASHBOARD_JWT_PUBLIC_KEY` env var code path — deferred.

**Phase requirement IDs (formalize during planning as MCPAUTH-04 + MCPAUTH-05):**
- **MCPAUTH-04** (candidate): User requires the `mcp.use` permission in their active org (or `admin.system` globally) to complete `/oauth/authorize` and open an MCP connection.
- **MCPAUTH-05** (candidate): Org admins (`admin.org`) can view and revoke DCR'd OAuth clients belonging to users within their own org via `/admin/clients`; cross-org operations return 403 + audit entry.

</domain>

<decisions>
## Implementation Decisions

### `mcp.use` permission model
- **D-01:** New permission `mcp.use` added to `packages/dashboard/src/permissions.ts ALL_PERMISSIONS` with group `"Administration"` (or new group `"MCP"` — planner decides). Label: `"Connect via MCP"`.
- **D-02:** **Per-org + global `admin.system` override.** `mcp.use` is evaluated against the user's CURRENT active org via `resolveEffectivePermissions(user, orgId)`. `admin.system` holders bypass the per-org check (matches existing admin.system semantics across the codebase).
- **D-03:** **Grant authority:** `admin.org` within their own org + `admin.system` across all orgs. Regular users (including those with write-level permissions) cannot grant `mcp.use` to themselves or others.
- **D-04:** **Backfill strategy:** New SQL migration (expected `054-add-mcp-use-permission`) adds one `user_permissions` (or equivalent — follow existing role/permission schema) row per existing `user × org` pair, explicitly granting `mcp.use`. Preserves day-1 continuity — every user who currently has any dashboard access keeps MCP access. **Users created after 31.2 ships do NOT inherit the permission** — explicit grant required via `/admin/users` or `/admin/roles`.
- **D-05:** **Wrong-org UX:** If the user has `mcp.use` in Org-A but their active session org is Org-B (no `mcp.use`), `/oauth/authorize` renders a "switch org" card: "You have MCP access in Org-A. Switch active org and retry?" with a **Switch** button that triggers a session org-swap + redirect back to the same `/oauth/authorize?...` URL. If the user has `mcp.use` in NO org, render a plain "you don't have MCP access — ask your org admin" card (no switch button).
- **D-06:** **Gate placement:** `/oauth/authorize` GET handler checks `mcp.use` immediately after `resolveEffectivePermissions` (after Phase 31.1 step 5 partitionScopes). Same check re-enforced on POST `/oauth/authorize/consent` (defense-in-depth, same pattern as 31.1 admin-scope gate). Admin-system bypass applies in both places.

### Tool visibility source
- **D-07:** **RBAC ∩ scope (defense-in-depth):** The MCP `tools/list` response is filtered by TWO layers composed sequentially:
  1. Phase 30.1 `filterToolsByScope(tools, scopes)` — keeps the 30.1 invariants (read → `.view` only; write adds `.create/.update/.manage/.delete/.admin.users/.admin.org`; empty-set on unknown sub).
  2. NEW `filterToolsByRbac(tools, userPermissions)` — every tool that requires a specific permission (declared via tool metadata; see D-09) must have its permission present in `userPermissions`, else filtered out.
  Final list = scope-filter output ∩ RBAC-filter output. A user with `scans.create` AND scope=`write` sees `dashboard_scan_site`; a user with `scans.create` AND scope=`read` does NOT (30.1 read-tier excludes write tools); a user WITHOUT `scans.create` AND scope=`write` does NOT (RBAC narrows further).
- **D-08:** **Tool execution time check:** Every tool handler still enforces `hasPermission(request, requiredPermission)` at invocation — last-line-of-defense even if a scope-filter drift sneaks a tool into the list. Matches Phase 30.1's "runtime iteration test" invariant.
- **D-09:** **Per-tool RBAC declaration:** Tool metadata (`packages/dashboard/src/mcp/metadata.ts` and friends) gains a `requiredPermission?: string` field. Read tools like `dashboard_list_reports` → `reports.view`; destructive `dashboard_scan_site` → `scans.create`; admin tools like `dashboard_create_user` → `users.create`; etc. Admin tools that previously relied on the `admin.system` scope map to their real underlying RBAC permission (e.g. `dashboard_update_org` → `admin.org`). Tools with no `requiredPermission` (pure info reads like health) stay visible to any mcp.use holder.

### OAuth scope retirement
- **D-10:** **`scopes_supported` metadata narrows to `['read', 'write']`.** Remove `admin.system`, `admin.org`, `admin.users` from the published `.well-known/oauth-authorization-server` metadata. A client requesting a removed admin.* scope on `/oauth/authorize` gets `invalid_scope` 400. No silent drop — the scopes simply don't exist at the OAuth layer anymore.
- **D-11:** **`scopes` JWT claim kept as informational.** Tokens continue to carry `scopes: ['read']` or `scopes: ['read', 'write']`. Used only for audit-log context and the Phase 30.1 scope-filter. Tool-call-time enforcement is RBAC, not scope.
- **D-12:** **`/oauth/authorize` scope validation updates:** `VALID_SCOPES` constant narrows to `{read, write}`. `partitionScopes` simplifies — no more admin.* branch (since admin.* can't be requested via OAuth anymore). Existing code path stays correct for `read`/`write` — those require no RBAC permission beyond `mcp.use` to grant.

### Existing-token handling
- **D-13:** **Silently ignore admin.* scope claims on verify.** Tokens minted before 31.2 may carry `scopes: ['read', 'write', 'admin.system', 'admin.org', 'admin.users']`. The RS256 verifier continues to accept them (legacy PEM path + JWKS path both). Phase 30.1 scope-filter receives the full scopes array, but since admin.* filter-rules are being removed (see D-14), those scope entries become no-ops. Effective behavior: user's RBAC governs tool visibility regardless of what scope bundle they have in their token. **Zero forced re-auth on deploy** — critical for the Claude Desktop clients already live from 31.1 smoke.
- **D-14:** **Phase 30.1 `tool-filter.ts` rule changes:** The `admin.system` scope-to-permission mapping becomes a no-op (the rule still exists for back-compat but maps to no additional tools because admin tools move to RBAC gating via D-07/D-09). `read` and `write` tier-mappings stay exactly as 30.1 defined. **No changes** to `role-repository.ts` `getUserPermissions` empty-set invariant.

### `client_credentials` grant removal
- **D-15:** **Dashboard `/oauth/token` removes `client_credentials` support.** Handler returns 400 `unsupported_grant_type` when called with `grant_type=client_credentials`. Dashboard AS is now exclusively for user flows (authorization_code + refresh_token).
- **D-16:** **Per-service `/api/v1/oauth/token` endpoints stay as-is** on compliance / branding / llm for service-to-service bootstrap via `ServiceTokenManager` (Phase 31.1 D-10 invariant preserved).

### Org-scoped DCR client management
- **D-17:** **`oauth_clients_v2` gains `registered_by_user_id TEXT NULL` column** via new migration (expected `055-oauth-clients-v2-user-link`). NULL for rows created during 31.1 before this phase landed. FK to `dashboard_users.id` ON DELETE SET NULL.
- **D-18:** **Back-fill timing — at FIRST consent.** DCR registration doesn't carry a user session (per RFC 7591 intent). The `/oauth/authorize/consent` POST handler sets `registered_by_user_id = user.id` on the client row if currently NULL (first consent wins). Subsequent consents from other users don't overwrite. Back-fill is a tiny UPDATE inside the existing consent transaction.
- **D-19:** **Visibility filter at `/admin/clients`:**
  - `admin.system` — sees ALL DCR clients (cross-org).
  - `admin.org` in Org-X — sees DCR clients WHERE `registered_by_user_id` belongs to a user whose active org is Org-X (or was Org-X at consent time — resolved via `dashboard_users` lookup). Rows with `registered_by_user_id IS NULL` (pre-D-18 backfill) are visible only to `admin.system`.
  - Non-admin users — `/admin/clients` continues to require `admin.org` OR `admin.system` to load at all. Regular users don't see the page.
- **D-20:** **Revoker = same RBAC as viewer.** If you can see the client row, you can click Revoke. No new `mcp.clients.revoke` permission. Revoke semantics:
  1. Sets `revoked_at = NOW()` on `oauth_clients_v2`.
  2. Cascades to `oauth_refresh_tokens.revoked_at` for every refresh chain on that client.
  3. The RS verifier adds a post-JWT client-status check: reject access tokens on next use if `oauth_clients_v2.revoked_at IS NOT NULL`. In-flight access tokens continue to work for up to their 1h TTL but cannot be refreshed.
  4. UI shows an immediate toast: "Client revoked. Existing sessions will disconnect within 1 hour; refresh is blocked immediately."
- **D-21:** **Cross-org revoke defense:** If `admin.org` in Org-A POSTs `/admin/clients/dcr/:clientId/revoke` for a client whose `registered_by_user_id` is in Org-B:
  1. Server returns **403** (not 404 — we want a clear audit trail).
  2. Writes an `agent_audit_log` row with `tool_name='admin.clients.cross_org_revoke_attempt'`, `outcome='denied'`, `outcome_detail=<clientId>:<attempted-by-user-id>:<viewer-org>:<client-org>`.
  3. UI never presents the Revoke button for cross-org rows (D-19 filter) — this branch defends against tampered POSTs / direct API calls.

### Service-side `WWW-Authenticate` parity (G9 closure)
- **D-22:** **Compliance, branding, and LLM MCP middleware emit `WWW-Authenticate`** on every 401 — mirrors the dashboard fix from 31.1 smoke commit `e0637ac`. Each service's `src/auth/middleware.ts` takes a new `resourceMetadataUrl` option (`${service.publicUrl}/.well-known/oauth-protected-resource`), emits `WWW-Authenticate: Bearer resource_metadata="<url>"` on missing-bearer 401, adds `, error="invalid_token"` on failed-verify 401. Tests updated in all three packages.
- **D-23:** **Shared middleware factory** — Claude's discretion whether to extract a shared `createMcpAuthPreHandler` into `@luqen/core/mcp/middleware` at this point, or keep three byte-identical factories per 31.1 D-34. Recommend reassessment after 31.2 lands because now all 4 services emit the same WWW-Authenticate pattern AND the same JWKS verifier AND the same aud enforcement — the duplication case weakens.

### Admin UI changes
- **D-24:** **`/admin/clients` template extends with:**
  - New "Revoked" column (shows `revoked_at` date when set, "—" otherwise).
  - Existing Kind column already distinguishes DCR vs admin-registered (Phase 31.1 Wave 4 work).
  - Revoke button hidden when `revoked_at IS NOT NULL` (already revoked).
  - Org column (shown to admin.system only — redundant for admin.org viewers since all rows are their own org).
- **D-25:** **`/admin/users` or `/admin/roles` UI knob for `mcp.use`** — implementation choice for the planner. Simplest: mcp.use appears in the permission list on the role editor (adding it to a role grants it to every user in that role). More granular: per-user checkbox on `/admin/users/:id`. Claude's discretion; recommended to add to the role editor (consistent with other permissions) + document that the migration back-fills it onto existing role assignments.

### Claude's Discretion
- Exact SQL schema: migration file numbering (054 onwards), column naming consistency with existing tables, whether to index `(registered_by_user_id, revoked_at)` for fast admin-clients list queries.
- UI copy for the consent-screen "switch org" CTA.
- Whether to add a tool-metadata validator test (grep-style) confirming every dashboard MCP tool has a `requiredPermission` declaration (D-09). Strong recommendation — prevents drift.
- Whether to extract the shared middleware factory (D-23) now or defer.
- Exact audit-event naming scheme for the cross-org defense (D-21 uses `admin.clients.cross_org_revoke_attempt`; could standardize as `auth.cross_org_access_denied` for forward generalizability).
- How to surface the "your client was revoked" UX in Claude Desktop — probably just let the 401 + re-auth flow kick in; explicit error-page work is nice-to-have.
- Whether to add a short `user_setup` readme for this phase documenting the new `mcp.use` permission for deployments.

</decisions>

<specifics>
## Specific Ideas

- Day-1 continuity matters: the Claude Desktop clients from 31.1 smoke (4 DCR'd `Claude` clients in dashboard.db) must keep working after 31.2 ships. The migration-backfill + silently-ignore-admin-scopes posture (D-04 + D-13) makes that automatic.
- Org admins should feel like first-class citizens on `/admin/clients` — they've been blocked from their own org's management today (the whole G1 gap). UI copy should make the org scope obvious ("Showing clients registered in {activeOrg}").
- Cross-org defense (D-21) is philosophically aligned with Phase 30.1 — empty-set on unknown sub, 403 + audit on hostile access. Same mental model.
- Per-tool `requiredPermission` declaration (D-09) is the clean way to express "which dashboard ability does this tool need" — also serves as self-documentation for anyone reading the MCP tool set.

</specifics>
