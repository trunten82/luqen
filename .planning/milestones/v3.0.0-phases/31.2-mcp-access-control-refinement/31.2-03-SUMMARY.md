## Accomplishments

- Added `filterToolsByRbac(tools, effectivePerms) → string[]` to `@luqen/core/mcp` alongside existing `filterToolsByPermissions` and `filterToolsByScope`. Semantics identical to `filterToolsByPermissions`; the distinct name makes the http-plugin intersection call-site intent explicit.
- Rewrote http-plugin's `ListToolsRequestSchema` handler: for authenticated end users (`ctx.permissions.size > 0`), the visible tool list is the **intersection** of scope-filter output and RBAC-filter output. For S2S callers (empty perms), the handler falls through to scope-only — preserving the Phase 30.1 invariant exactly.
- Added a per-tool runtime RBAC guard via `CallToolRequestSchema` override (D-08 last-line-of-defense). The guard captures the SDK's default handler from `protocol._requestHandlers` BEFORE override, checks `meta.requiredPermission` against `ctx.permissions`, and either (a) returns an MCP error envelope with `isError: true` and a `Forbidden` message when the caller lacks the permission, or (b) delegates to the captured SDK handler on authorized paths — preserving all SDK-native dispatch semantics (input validation, output schema, task handling).
- Annotated the `admin.system` scope-filter rule with a Phase 31.2 D-14 comment — making explicit that `admin.*` scope values are no-ops post-31.2 (they cannot be newly minted because `scopes_supported` narrowed to `['read','write']`, but the rule persists for pre-31.2 back-compat tokens).
- Added a D-14 lock-in test (`filterToolsByScope` block) confirming pre-31.2 legacy scope bundles (`['admin.system','admin.org','admin.users','read','write']`) return the SAME tool list as `['read','write']` — admin.* scope values cannot privilege-escalate.
- Added a 5-test drift-prevention suite (`tests/mcp/tool-metadata-drift.test.ts`) that catches any future dashboard MCP tool lacking `requiredPermission`, typo'd permission ids, or drift between `DASHBOARD_TOOL_METADATA` and `createDashboardMcpServer`'s `_registeredTools` registry.

## Composition Rule (Post-Edit)

```typescript
// packages/core/src/mcp/http-plugin.ts
mcpServer.server.setRequestHandler(ListToolsRequestSchema, async () => {
  const ctx = getCurrentToolContext();
  let allowedNames: readonly string[];
  if (ctx == null) {
    allowedNames = [];
  } else if (ctx.permissions.size > 0) {
    // (a) End user with RBAC — D-07 intersection
    const byScope = new Set(filterToolsByScope(toolMetadata, ctx.scopes));
    const byRbac  = new Set(filterToolsByRbac(toolMetadata, ctx.permissions));
    allowedNames = [...byScope].filter((name) => byRbac.has(name));
  } else {
    // (b) S2S caller — scope-filter alone (Phase 30.1 invariant)
    allowedNames = filterToolsByScope(toolMetadata, ctx.scopes);
  }
  // ... _registeredTools projection unchanged ...
});
```

## 30.1 Invariant Preservation Proof

**S2S empty-perms branch:** The else-branch runs when `ctx.permissions.size === 0`. Intersecting `filterToolsByRbac` on an empty permissions Set would return only tools with `requiredPermission == null` (the unannotated-passthrough rule). For a dashboard MCP server where every tool has a `requiredPermission`, that would mean zero tools visible — breaking every live service-to-service bootstrap token that authenticates with the `client_credentials` grant.

Instead, when `permissions.size === 0`, the handler falls through to `filterToolsByScope(tools, scopes)` alone. This matches the Phase 30.1 exact behaviour and is verified by new test **Test F** in `http-plugin.test.ts`:

```
Test F (30.1 non-regression): S2S caller w/ empty perms + scope=read → scope-filter is authoritative
  expect(names).toContain('dashboard_list_reports');   // read-tier ✓
  expect(names).toContain('dashboard_info');            // unannotated ✓
  expect(names).not.toContain('dashboard_scan_site');   // write-tier ✗ (scope-filter excluded)
```

**T-31.2-03-07 (Spoofing — S2S caller asserts empty permissions to bypass RBAC):** ctx.permissions is populated by the service's auth middleware via `resolveEffectivePermissions`, not by the client. A token cannot control the permissions attached to its request — so the empty-perm branch is reached only when the authoritative resolver returns an empty set (legitimate S2S caller), never via client tampering.

## Runtime Guard Code (Post-Edit)

```typescript
// packages/core/src/mcp/http-plugin.ts
const protocolAsAny = mcpServer.server as unknown as {
  _requestHandlers?: Map<string, (request: unknown, extra: unknown) => Promise<unknown>>;
};
const sdkCallToolHandler = protocolAsAny._requestHandlers?.get('tools/call');

mcpServer.server.setRequestHandler(CallToolRequestSchema, async (request, extra) => {
  const ctx = getCurrentToolContext();
  const toolName = request.params.name;
  const meta = toolMetadata.find((t) => t.name === toolName);
  if (ctx != null && ctx.permissions.size > 0 &&
      meta?.requiredPermission != null &&
      !ctx.permissions.has(meta.requiredPermission)) {
    return {
      content: [{ type: 'text', text: JSON.stringify({
        error: 'Forbidden',
        message: `Tool "${toolName}" requires permission "${meta.requiredPermission}"`,
        requiredPermission: meta.requiredPermission,
      }) }],
      isError: true,
    };
  }
  if (sdkCallToolHandler == null) {
    throw new McpError(ErrorCode.MethodNotFound, 'tools/call handler not initialised on this McpServer');
  }
  return (await sdkCallToolHandler(request, extra)) as never;
});
```

## Decisions Made

- **filterToolsByRbac distinct from filterToolsByPermissions** — semantically identical, but a distinct name makes http-plugin's intersection call-site intent explicit without risking silent breakage for legacy non-http-plugin call sites that still use the old name.
- **Intersection branch only when ctx.permissions.size > 0** — S2S callers (empty perms) fall through to scope-only to preserve the Phase 30.1 invariant. Intersecting an empty Set of permissions with scope would always yield zero annotated tools, breaking every live S2S bootstrap path.
- **CallToolRequestSchema override delegates to SDK-native handler** — captured via `protocol._requestHandlers.get('tools/call')` BEFORE the override. Preserves input validation, output schema, task handling, and error wrapping exactly. Safer than reimplementing dispatch.
- **Guard error envelope includes the missing permission name** — T-31.2-03-04 accepted risk: the permission name is not privileged and helps legitimate clients self-diagnose.
- **admin.system scope-rule kept as `return false` not deleted** — pre-31.2 tokens carrying admin.* scope values pass through as no-ops rather than causing privilege escalation. Kept for defence; its effective behaviour is already a no-op.
- **Task 3 is a lock-in test, not a backfill** — the Discovery Finding confirmed all 19 dashboard tools already have `requiredPermission`. The drift test is a CI tripwire against future drift; no RED-to-GREEN code change needed in the src/ tree.
