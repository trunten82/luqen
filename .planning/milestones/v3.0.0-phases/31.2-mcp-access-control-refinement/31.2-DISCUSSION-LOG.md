## mcp.use permission model

### Q: Where does `mcp.use` live in the RBAC hierarchy?

| Option | Description | Selected |
|--------|-------------|----------|
| Per-org only | Evaluated against active org only; no admin.system bypass |  |
| Per-org + global `admin.system` override | Per-org default + system admins bypass | ✓ |
| Global only | Simple flag on the user |  |

**User's choice:** Per-org + global `admin.system` override

### Q: Which existing roles get `mcp.use` by default after migration?

| Option | Description | Selected |
|--------|-------------|----------|
| Admin roles only | admin.system + admin.org holders auto-granted |  |
| Admin + 'developer' roles | Broader write-level role cluster |  |
| No default — explicit grant only | Disruptive day-1 |  |
| Everyone (existing users) | Preserves 31.1 behavior exactly |  |
| **User-specified:** explicit grant model going forward, BUT back-fill onto all CURRENTLY existing users via migration | Hybrid — day-1 continuity + new-users-explicit | ✓ |

**User's choice:** Back-fill onto all currently existing users; going forward, explicit grant required for new users (effectively chosen option 4 for the migration step + no-default for new accounts).

### Q: Who can grant/revoke `mcp.use`?

| Option | Description | Selected |
|--------|-------------|----------|
| admin.org within own org + admin.system everywhere | Matches existing admin.org pattern | ✓ |
| admin.system only | Too centralized |  |
| Via roles only (no per-user toggle) | Long-term clean; bigger UI |  |

**User's choice:** admin.org within own org + admin.system everywhere

### Q: Wrong-org UX — user has mcp.use in Org-A, active session is Org-B.

| Option | Description | Selected |
|--------|-------------|----------|
| Block with a 'switch org' CTA | One-click fix; preserves per-org invariant | ✓ |
| Auto-switch | Low friction; surprising |  |
| Plain denial | Simplest; worst UX |  |

**User's choice:** Block with a 'switch org' CTA

## Org-scoped client revocation

### Q: How should `/admin/clients` filter DCR client visibility?

| Option | Description | Selected |
|--------|-------------|----------|
| admin.system sees all; admin.org sees own-org only | Matches RBAC hierarchy | ✓ |
| admin.system all; admin.org own-org + own clients (cross-org personal) | Small convenience, OR clause |  |
| admin.system only (status quo) | Rejects G1 — invalid |  |

**User's choice:** admin.system sees all; admin.org sees own-org only

### Q: When is `registered_by_user_id` populated?

| Option | Description | Selected |
|--------|-------------|----------|
| At FIRST consent (not at DCR) | Matches RFC 7591 pre-auth intent. Recommended. | ✓ |
| At DCR registration if session cookie present | More accurate for 'Add Connector while logged in' |  |
| Track per-consent (user × client pairs) | Bigger UI rework |  |

**User's choice:** At FIRST consent

### Q: Who can revoke, and what does revoke kill?

| Option | Description | Selected |
|--------|-------------|----------|
| Revoker = same RBAC as viewer; hard-kills refresh + 401s access on next use | Simplest; expected UX. Recommended. | ✓ |
| Revoker = admin.system only | Asymmetric with visibility |  |
| New `mcp.clients.revoke` permission | Marginal benefit |  |

**User's choice:** Revoker = same RBAC as viewer; revoke hard-kills refresh + 401s in-flight access

### Q: Cross-org revoke attempts (admin.org in Org-A POSTs revoke for Org-B client).

| Option | Description | Selected |
|--------|-------------|----------|
| 403 + audit entry | Clear defense-in-depth. Recommended. | ✓ |
| 404 (hide existence) | Gentler info disclosure |  |
| Silent 200 no-op | Worst UX |  |

**User's choice:** 403 + audit entry
