# Phase 32.1 — Agent Chat UI: UAT Gap Closure

Close 9 gaps found during Phase 32 UAT on lxc-luqen. Three are blockers (agent unusable without them), three are major UX issues, three are cosmetic.

## Goal

Make the Phase 32 agent chat drawer usable end-to-end for both global dashboard admins and org-scoped users, with real tool execution, a way to start new conversations, and reactive UI updates on settings changes.

## Plans (ordered by dependency)

### Plan 01 — Global admin chat context (blocker) [Gap #1]
**Scope:** Let admin.system users without an org membership use the chat. Introduce a synthetic per-admin org namespace so conversation scoping still works.
**Changes:**
- `src/routes/agent.ts` — when `user.currentOrgId === undefined` AND user has `admin.system`, use `__admin__:${user.id}` as the effective orgId for all 5 routes (message, stream, panel, confirm, deny).
- `src/agent/agent-service.ts` — same fallback for orgId when calling `resolveAgentDisplayName` (returns default "Luqen Assistant" since synthetic orgs have no row).
- Tests: add 2 test cases to `tests/routes/agent.test.ts` — global admin no-org POST creates synthetic-org conversation; same user's messages don't leak to a real org.
**Acceptance:** Global admin can open the drawer and send a message end-to-end.

### Plan 02 — Wire in-process tool dispatcher (blocker) [Gap #2 + Gap #3]
**Scope:** Replace `ToolDispatcher({tools: []})` with handlers pulled from the MCP server's registered tools. Also populates the AgentService toolCatalog for LLM-visible descriptions.
**Changes:**
- `src/server.ts` — build an MCP server instance at boot (via `createDashboardMcpServer`) before AgentService construction; extract `_registeredTools` to derive:
  1. `toolCatalog: Record<name, {description, inputSchema}>` — pass to AgentService.
  2. `tools: AgentTool[]` — each tool's handler wraps the registered MCP callback.
- `src/agent/agent-service.ts` — no changes (already accepts `toolCatalog` after commit 7712e8c).
- Tests: add `tests/agent/tool-dispatch-integration.test.ts` covering a full dispatch for `dashboard_list_reports`.
**Acceptance:** Model calls `dashboard_list_reports`, tool executes, result row contains real report list, model summarises the list in its follow-up turn.

### Plan 03 — New-conversation control (major) [Gap #4]
**Scope:** Drawer header "New chat" button that clears localStorage, empties the messages region, and starts a fresh conversation on next message.
**Changes:**
- `src/views/partials/agent-drawer.hbs` — add `<button data-action="newChat" aria-label="…">` in drawer header, next to the close control.
- `src/static/agent.js` — click handler: clear `localStorage.luqen.agent.conversationId`, clear form's `data-conversation-id` attribute + hidden field, clear messages region, remove any active EventSource, show empty-state greeting (fetch `/agent/panel` with empty conversationId).
- 6 locale JSON files — new `agent.newChat.label`, `agent.newChat.confirm` keys.
- Tests: vitest smoke assertion for the new button + localStorage clear behaviour.
**Acceptance:** Click "New chat" → drawer empties, greeting shows, next message creates a new conversation row.

### Plan 04 — Reactive display-name update in drawer (major) [Gap #5]
**Scope:** Saving the org agent display name reflects immediately in the drawer without a hard refresh.
**Changes:**
- `src/views/partials/agent-drawer.hbs` — wrap the display name in `<span id="agent-display-name">`.
- `src/routes/admin/organizations.ts` — on successful POST to `/settings`, include an HTMX OOB swap `<span id="agent-display-name" hx-swap-oob="true">{new name}</span>` in the response body alongside the form re-render.
- Fallback: JS listener on `htmx:afterRequest` for the org-settings form path — reads the submitted value, updates `#agent-display-name` textContent directly.
**Acceptance:** Save "Luna" → drawer header shows "Luna" within the same page state, no reload required.

### Plan 05 — Capabilities tab layout fix (minor/cosmetic) [Gap #6]
**Scope:** Badges and long text on /admin/llm stop overflowing their containers.
**Changes:**
- `src/static/style.css` — the banner block `/* ---- Agent Admin (Phase 32) ---- */` already exists; add `flex-wrap: wrap` on the badge row container, `overflow-wrap: anywhere` on the card body.
- Test: a screenshot/css assertion in vitest (or Playwright if added) that the badge row wraps at narrow widths.
**Acceptance:** At 360px / 768px / 1280px viewport, no horizontal overflow on agent-conversation row or agent-system default panel.

### Plan 06 — Markdown rendering in assistant bubbles (cosmetic) [Gap #7]
**Scope:** Render common markdown (bold / italic / bullets / paragraphs) safely; raw text still flows through.
**Changes:**
- `src/static/agent.js` — on `done` frame, re-render the current streaming bubble: parse accumulated text via a small purpose-built renderer (no innerHTML; build DOM via `createElement` + `textContent`). Supported subset: `**…**` → `<strong>`, `*…*` → `<em>`, leading `- ` → `<ul><li>`, blank-line → `<p>`.
- Tests: unit tests for the renderer covering escape cases (literal asterisks, mixed content).
**Acceptance:** Assistant responses render bold, lists, paragraph breaks. Malicious payloads like `<script>` remain escaped (content enters via textContent).

### Plan 07 — Org settings navigation (minor) [Gap #8]
**Scope:** /admin/organizations/:id/settings reachable via existing UI.
**Changes:** add a "Settings" tab or link on the org detail page at `/admin/organizations/:id`.
- `src/views/admin/organization-detail.hbs` (or equivalent) — add `<a href="/admin/organizations/{{id}}/settings">{{t 'admin.organizations.settings.navLabel'}}</a>`.
- 6 locale files — new `admin.organizations.settings.navLabel` key.
**Acceptance:** User can click from the org detail page into the settings page.

### Plan 08 — Persist assistant tool_call row (minor) [Gap #9]
**Scope:** Replace the current window-shim workaround with a proper persisted assistant(tool_calls) row.
**Changes:**
- `src/agent/agent-service.ts` `dispatchAndPersist` — before dispatching the tool batch, persist ONE `role='assistant'` row with `toolCallJson` containing the full batch as JSON array.
- `windowToChatMessages` — drop the synthetic shim branch when a persisted assistant(tool_calls) row is present; keep the shim as a fallback for pre-32.1 conversations.
- Tests: regression test that sends a tool-chain message and reads the window back, asserts the assistant+tool pair exists as two rows with matching toolCallJson.
**Acceptance:** New conversations show an assistant row per tool batch; reading the window no longer synthesises shims for post-32.1 conversations.

## Ordering / dependencies

- Plan 01 is independent — can land first.
- Plan 02 is the critical unblocker — unlocks tests 6-9.
- Plan 03–07 are independent of each other; any order works.
- Plan 08 is a quality improvement; can land last.

Suggested execution order: 09 → 01 → 02 → 03 → 04 → 05 → 07 → 06 → 08. (CI green first.)

### Plan 09 — Green the CI pipeline [Gap #10]
**Scope:** Fix the 8 pre-existing test failures that have been red on every CI build since Phase 32-04 landed. All are documented in `32-agent-service-chat-ui/deferred-items.md` as unrelated to phase 32 code; they're stale expectations that need updating.
**Changes:**
- `tests/mcp/data-tools.test.ts` — 2 failures. Tests use `scopes: ['read']` but the tools require write-tier scopes post-Phase 31.2 scope filter. Update fixtures to pass `scopes: ['write']` or `['admin']` where the required permission is write/admin-tier.
- `tests/mcp/admin-tools.test.ts` — 3 failures (tool-count mismatches: 19 / 4 / 9). Same scope-filter root cause; update scope fixtures to match the intersection logic (Phase 31.2 D-07).
- `tests/mcp/http.test.ts` — 1 failure. Same scope fix.
- `tests/e2e/auth-flow-e2e.test.ts` — 2 failures. Tests assert `location === '/login'` but Phase 31.1 Plan 02 added `returnTo` query param: `'/login?returnTo=%2Fhome'`. Update assertions to `location.startsWith('/login')` or match the full path with returnTo.
- CI workflow — optional: add `npm test -w packages/llm` + `npm test -w packages/compliance` to the matrix if not already present, to catch future regressions faster.
**Acceptance:** `gh run list --branch master --limit 1` shows conclusion:success on the next pushed commit.
