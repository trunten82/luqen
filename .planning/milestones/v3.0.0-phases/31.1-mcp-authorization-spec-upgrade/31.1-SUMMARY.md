## Architectural posture after 31.1

- Dashboard is the single OAuth 2.1 Authorization Server for the Luqen platform
- Compliance / branding / LLM services are RS â€” they verify RS256-signed JWTs via the dashboard's `/oauth/jwks.json` with cache + kid-rotation
- Per-service audience indicators (RFC 8707) â€” token `aud` must include the target service's MCP URL or the RS rejects it
- External clients (Claude Desktop, MCP Inspector, IDE plugins) DCR themselves; open DCR with user-consent gate; rate-limited to 10/hr/IP
- PKCE S256 required on every authorization_code flow (no `plain`)
- Access tokens 1h; refresh tokens rotate-on-use with reuse detection + 30d absolute lifetime; reuse triggers chain revocation + `oauth.refresh_reuse_detected` audit row
- Signing keys rotate via overlap window; quarterly auto-rotation cron + admin "Rotate now" button; old keys retire only after every token signed with them has expired
- Legacy `DASHBOARD_JWT_PUBLIC_KEY` env var honored with `console.warn` during transition â€” full removal deferred
