# Phase 31.1: MCP Authorization Spec Upgrade — Context

**Gathered:** 2026-04-19
**Status:** Ready for planning

<domain>
## Phase Boundary

**What this phase delivers:**
- Dashboard becomes an **OAuth 2.1 Authorization Server** exposing:
  - `GET  /oauth/authorize` — Authorization Code + PKCE browser flow with consent screen
  - `POST /oauth/token` — token endpoint supporting `authorization_code`, `refresh_token`, and the existing `client_credentials` grant
  - `POST /oauth/register` — open Dynamic Client Registration (RFC 7591)
  - `GET  /.well-known/jwks.json` — JWKS endpoint with RSA public key set + rotation support
  - `GET  /.well-known/oauth-authorization-server` — AS metadata per RFC 8414
- Each Luqen service (compliance, branding, LLM, dashboard-as-RS) publishes:
  - `GET /.well-known/oauth-protected-resource` per RFC 9728, pointing to the dashboard as AS
  - Audience enforcement — tokens are accepted only if `aud` includes the service's MCP URL
- RS256 JWT verifier migration: services stop reading `DASHBOARD_JWT_PUBLIC_KEY` env var and pull keys from the dashboard's JWKS endpoint (with TTL cache)
- New DB tables on `dashboard.db`: `oauth_clients_v2` (DCR registrations), `oauth_authorization_codes` (short-lived), `oauth_refresh_tokens` (rotation chain + reuse detection), `oauth_user_consents` (remembered per-client consents), `oauth_signing_keys` (current + retiring keys for rotation)
- `/admin/clients` page extended: show DCR'd clients alongside admin-registered ones, with revoke and "connected users" visibility
- Consent screen UI at `/oauth/authorize` — HTMX-rendered, matching dashboard design system

**What this phase does NOT deliver:**
- Phase 32's internal dashboard-to-MCP auth path — that stays cookie-session + server-minted per-user JWT (Phase 32 D-03/D-04) and is unaffected by this phase
- External-AS integration (Keycloak, Auth0) — single dashboard AS, full stop
- Per-org AS segregation (each org runs its own AS) — deferred as future work
- User-facing `/account/connected-apps` page — admin `/admin/clients` is the only UI in MVP (user-facing view routes to Phase 33 if demanded)
- A2A peer registration flows — v3.1+ (MCPE-03 in REQUIREMENTS)
- Device Code grant (RFC 8628) — deferred; external clients for this MVP use Authorization Code + PKCE
- Deprecation of compliance/branding/LLM `/api/v1/oauth/token` endpoints — they keep running untouched for internal `ServiceTokenManager` bootstrap

**Phase requirement IDs:** MCPAUTH-01, MCPAUTH-02, MCPAUTH-03 (formalized in REQUIREMENTS.md 2026-04-19).

<decisions>
## Implementation Decisions

### Authorization Server topology
- **D-01:** **Single AS in the dashboard.** Dashboard exposes `/oauth/authorize`, `/oauth/token`, `/oauth/register`, `/oauth/jwks.json`, and the AS `.well-known` metadata. Compliance, branding, and LLM services are pure **Resource Servers** — they verify tokens against the dashboard's JWKS. One MCP client registration covers all services.
- **D-02:** The **dashboard itself is both AS and RS**. Its own MCP endpoint (`POST /mcp`) verifies tokens the AS issues, same as the other services. No special internal loopback — the dashboard RS side calls its own JWKS the same way the other services do.
- **D-03:** **No external AS integration** (Keycloak, Auth0, etc.). Matches REQUIREMENTS' "admins in control through the dashboard, not config files" and keeps self-hosted Luqen deployments operationally contained.

### Audience model (RFC 8707 resource indicators)
- **D-04:** **Per-service audience indicators.** Client sends `resource=https://<service>.luqen.example/mcp` on the Authorization Code request (RFC 8707). Dashboard mints a token with `aud=[that resource URL]`. Each service's RS256 verifier checks `aud` contains the service's own MCP URL; if not, returns 401 before any tool dispatch.
- **D-05:** **Multi-service resource bundle.** A client that needs multiple services (e.g. Claude Desktop wanting compliance + branding + LLM + dashboard) requests multiple `resource` parameters on a single Authorization Code request; dashboard mints a token with `aud` = all requested services. Simplifies the external-client UX (one OAuth flow covers everything they need).
- **D-06:** **Dashboard chat agent path** — Phase 32's `AgentService` doesn't use the external OAuth flow; it mints short-lived per-service JWTs at tool-dispatch time with `aud=<target service>` derived from the tool's origin (via `@luqen/core/mcp` metadata). This is internal, server-side, and reuses the dashboard JWT signing key — same key material, same verifier, separate minting path from the external OAuth endpoints.

### Scope vocabulary
- **D-07:** **Reuse existing scope suffixes.** Scope values issued by the AS are the same Phase 30.1 scope-filter rules consume: `read`, `write`, `admin.system`, `admin.org`, `admin.users`. No new scope namespace. No changes to `filterToolsByScope` / `filterResourcesByScope`.
- **D-08:** **Scope whitelist at /oauth/authorize.** Clients may request `scope=read`, `scope="read write"`, or `scope="read write admin.org"` — dashboard rejects unknown scope strings with `invalid_scope`. The whitelist is derived from the Phase 30.1 scope-to-permission map (`read` → `.view`, `write` → `.create/.update/.manage/.delete/.admin.users/.admin.org`, `admin.system` → `.admin.system`).
- **D-09:** **Admin-system scope requires admin-system RBAC.** User authorizing a client that requests `admin.system` must themselves have `admin.system` permission in their active org (via `resolveEffectivePermissions`); otherwise the consent screen shows a friendly "This client requests admin access you don't have" card and the flow cannot complete. Prevents non-admin users from delegating admin via a malicious client.

### Grant coexistence
- **D-10:** **Keep compliance/branding/LLM `/api/v1/oauth/token` as-is.** These continue to serve `client_credentials` grants for internal `ServiceTokenManager` bootstrap (`packages/dashboard/src/auth/service-token.ts`). Phase 31.1 does NOT touch them.
- **D-11:** **Dashboard `/oauth/token` supports three grants:** `authorization_code`, `refresh_token` (new in 31.1), and `client_credentials` (re-exposed on the dashboard for symmetry; the existing service-level `/api/v1/oauth/token` endpoints stay untouched). External MCP clients use `authorization_code` + `refresh_token`. Any internal component that wants a user-less service token can still use `client_credentials` against a pre-registered admin OAuth client.

### Org binding
- **D-12:** **Token's `orgId` claim = user's active dashboard session org** at the moment `/oauth/authorize` is completed. If the user switches orgs in the dashboard later, existing tokens stay scoped to the org they were minted for (until refresh, where `orgId` is reasserted from the user's then-current session). Matches the dashboard UX — users operate in one org at a time.
- **D-13:** **No org picker in consent screen.** Keeps the MVP consent UI simple. Multi-org users who want a token for a different org must switch org in the dashboard first, then re-authorize. Flagged for possible Phase 33 UX improvement (see Deferred).
- **D-14:** **`orgId` claim is user-sourced, not client-sourced** — clients cannot request `orgId` as a parameter on `/oauth/authorize`. The invariant "no `orgId` in tool `inputSchema`" (Phase 29 D-13) extends to the OAuth layer: tokens carry the user's org; tools source orgId from the token context.

### Parallel clients per user
- **D-15:** **Unlimited active MCP clients per user.** Each `client_id` has its own authorization record; a user may have Claude Desktop + MCP Inspector + an IDE plugin + custom scripts all authorized simultaneously. Admin `/admin/clients` surfaces a revoke button per client. No soft cap or prompt in MVP.

### Dynamic Client Registration
- **D-16:** **Open DCR per RFC 7591.** `POST /oauth/register` accepts any well-formed `client_metadata` body (name, redirect_uris, grant_types=[authorization_code, refresh_token], token_endpoint_auth_method=[none | client_secret_basic], scope, software_id, software_version) and returns `{client_id, client_secret?, registration_access_token?, registration_client_uri?}`. No admin approval. Safety depends on the user-consent gate in Authorization Code flow (no token without explicit user Approve) + admin-visible client revocation.
- **D-17:** **Rate-limit DCR aggressively.** `@fastify/rate-limit` on `POST /oauth/register` at 10 req/hour/IP (with `onSend` JSON hook per `feedback_rate_limiter.md`). Defends against registration spam that would pollute `/admin/clients`.
- **D-18:** **Public (PKCE-only) and confidential clients both supported.** `token_endpoint_auth_method: 'none'` → public client, PKCE required, no `client_secret` returned. `token_endpoint_auth_method: 'client_secret_basic'` → confidential client, `client_secret` returned once in registration response. Both paths require PKCE S256.

### Consent UX
- **D-19:** **Full consent on first connect per `client_id`.** The `/oauth/authorize` endpoint renders a consent screen showing: client name, requested scopes (human-readable descriptions), requested resource URIs, requested redirect URI (so user can spot a rogue one). User clicks Allow or Deny. On Allow, a row is inserted into `oauth_user_consents` with `(user_id, client_id, scopes, resources, consented_at)`.
- **D-20:** **Remember consent for same-or-narrower scopes.** Subsequent `/oauth/authorize` calls from the same `(user_id, client_id)` auto-approve if requested scopes ⊆ consented scopes AND requested resources ⊆ consented resources. If the client requests a new scope or resource, the consent screen re-appears showing "New permissions requested" highlighted; user re-approves to extend the consent.
- **D-21:** **All-or-nothing scope approval.** No per-scope checkboxes. Consent screen has Allow (for the full bundle) or Deny. Matches MCP client reality — clients request what they need; the user makes one call.
- **D-22:** **Denial behavior.** Clicking Deny redirects to the client's `redirect_uri` with `error=access_denied` per OAuth 2.1 §4.1.2.1 — no token minted, no consent row written.
- **D-23:** **Consent screen design** — native Luqen layout (no new CSS framework), uses existing dashboard design system classes per `feedback_design_system_consistency.md`. Approve button uses the existing primary action color; Deny uses the existing destructive/secondary color. Must pass pa11y (the agent's own product) — AA contrast, keyboard navigable, labels on all form controls.

### Signing keys + rotation
- **D-24:** **JWKS endpoint at `/.well-known/jwks.json`.** Lists the dashboard's current signing key(s) in JWK format with `kid` (key id), `kty=RSA`, `alg=RS256`, `use=sig`. Supports `Cache-Control: public, max-age=3600` — services cache for 1h, auto-refresh on expiry or on `kid` miss.
- **D-25:** **Key rotation via overlap window.** On rotation: generate new key, add to JWKS alongside old, mint all new tokens with new `kid`. Old key stays in JWKS until it's been `access_token_max_lifetime + refresh_token_max_lifetime = 30d + 1h` since the last token was issued with it — then removed. Enforced via `oauth_signing_keys` table with `created_at`, `retired_at`, `removed_at` columns. A cron job (existing scheduler pattern) sweeps daily.
- **D-26:** **RSA 2048-bit minimum.** Use Node's `crypto.generateKeyPairSync('rsa', {modulusLength: 2048})`. Key material stored in `oauth_signing_keys` table, PEM-encoded. Private key encrypted at rest using the same encryption pattern as `service_connections` (v2.8.0).
- **D-27:** **Deprecate `DASHBOARD_JWT_PUBLIC_KEY` env var for services.** After 31.1 ships, services ignore the env var and pull from JWKS only. The env var remains defined for backwards-compat during the transition deploy but emits a `console.warn` at startup if set. Planner decides whether to fully remove in 31.1 or leave the warning path in for a phase or two (Claude's discretion).

### Token lifetimes
- **D-28:** **Access token: 1 hour (3600s).** Standard OAuth 2.1 default. Ship as a non-configurable constant in MVP (add env var if a deployment demands tighter — Claude's discretion).
- **D-29:** **Refresh token: rotate-on-use + reuse detection + 30d absolute lifetime.** Every `grant_type=refresh_token` call returns a NEW refresh token and marks the old one `rotated=true`. If a rotated token is presented again, **all refresh tokens in that chain are revoked** (revoke = delete from `oauth_refresh_tokens`), access tokens issued from them become invalid at next verification pass (cache TTL), and the user must re-authorize via `/oauth/authorize`. 30-day absolute lifetime from the original issuance (not per-rotation).
- **D-30:** **Authorization code: 60s lifetime, single use.** Codes are stored in `oauth_authorization_codes` with TTL; consumed on `/oauth/token` exchange, then deleted. Presenting a consumed code returns `invalid_grant`.

### PKCE
- **D-31:** **S256 only.** Authorization endpoint responds 400 `invalid_request` with `error_description="code_challenge_method must be S256"` if the client sends `code_challenge_method=plain` or omits it. All confirmed MCP clients (Claude Desktop, MCP Inspector, SDKs) support S256. Simpler attack surface.
- **D-32:** **PKCE required for all authorization_code flows.** Both public and confidential clients must include `code_challenge`; the token endpoint rejects the code exchange if `code_verifier` is missing or fails the S256 comparison. Aligns with OAuth 2.1 §4.1.1.

### Reuse of existing dashboard RS256 verifier
- **D-33:** **`packages/dashboard/src/mcp/verifier.ts` evolves.** Today it hardcodes `importSPKI(pem, 'RS256')` from a single env PEM. Phase 31.1 swaps this for a `createRemoteJWKSet(new URL(JWKS_URI))` from `jose`, with caching built in. Verifier keeps the RS256-only allowlist and the payload shape (`sub`, `scopes`, `orgId`, `role`, `iat`, `exp`). Add `aud` claim verification to enforce per-service audience.
- **D-34:** **Same verifier pattern propagates to compliance/branding/LLM.** Each of those services currently has an equivalent RS256 verifier in its own `mcp/verifier.ts` (check existing code during planning). All three swap to the JWKS-backed variant. Shared verifier factory may be worth extracting into `@luqen/core/mcp/verifier.ts` — Claude's discretion.

### Claude's Discretion
- Exact table schemas for `oauth_clients_v2`, `oauth_authorization_codes`, `oauth_refresh_tokens`, `oauth_user_consents`, `oauth_signing_keys` — follow migration 047/048 style (see 31-CONTEXT.md).
- Whether to retire the existing `oauth_clients` table (used by current bootstrap) or add `v2` and run both in parallel — planner picks based on data-migration cost.
- Whether to extract a shared verifier factory into `@luqen/core` or keep per-service verifier files.
- How the admin `/admin/clients` page distinguishes DCR'd clients from admin-registered ones (column badge, separate sections, etc.).
- Consent-screen exact copy + iconography (respect design-system classes).
- Whether `registration_access_token` is issued with DCR to let clients update/delete their own registrations (RFC 7592) — useful but adds surface; defer to planner judgment.
- Testing harness: reuse `role-repository.test.ts`-style in-memory SQLite for repo tests; plan integration tests that run an actual OAuth flow against a test server (MCP Inspector is a good reference client).
- Consent-screen rate limiting (CSRF tokens, prevent POST-only forgery).
- Whether refresh token reuse detection persists the revoke event to `agent_audit_log` with a synthetic `tool_name='oauth.refresh_reuse_detected'` — useful for incident response; Claude's discretion.

</decisions>

<specifics>
## Specific Ideas

- Consent screen mental model: "Claude Desktop wants to sign in as you and access: scan reports (read), run scans (write), view brand scores (read). You can revoke this anytime from /admin/clients." Approve / Deny.
- JWKS rotation cadence: quarterly by default (cron in `scheduler.ts`), plus an admin-triggered "rotate now" button on a new `/admin/oauth-keys` page (may be a sub-tab on `/admin/clients`).
- Claude Desktop is the primary validation target — it handles open DCR + Authorization Code + PKCE + refresh tokens natively (per MCP spec). Verify flows end-to-end against it during execution, not just against MCP Inspector.
- Audit event for refresh-token reuse detection: write to `agent_audit_log` with `tool_name='oauth.refresh_reuse_detected'`, `outcome='error'`, `outcome_detail='chain_revoked client_id=<X>'`. Reuses the Phase 31 audit table — no new schema.
- Consent-remembered invariant: adding a new scope to the bundle forces re-consent; removing a scope silently auto-approves (user granted a superset). Narrowing is safe; widening requires consent.

</specifics>

<deferred>
## Deferred Ideas

### Deferred to Phase 32 (explicit non-goal here)
- Internal dashboard agent auth (cookie session + server-minted JWT) — Phase 32 D-03 owns this path independently.

### Deferred to Phase 33 or later
- **User-facing `/account/connected-apps` page** where a user can see which MCP clients they've authorized and revoke individual grants (complements admin view). Phase 33 UX enhancement.
- **Org picker in consent screen** for multi-org users who want a token for a non-active org. Requires a "re-authorize with different org" UX.
- **Consent-scope granularity UI** (per-scope checkboxes, tiered tiers). Defer until users complain; all-or-nothing is fine for v3.0.
- **Device Code grant (RFC 8628)** — enables headless CLI tools without a browser. Matches a real MCP use case but adds complexity; v3.1 backlog.

### Deferred to v3.1+ milestones
- **External AS federation** (SAML / OIDC upstream) — single-sign-on from corporate IdP. Out of scope for a self-hosted MVP.
- **Token introspection endpoint (RFC 7662)** — services currently self-validate via JWT; introspection needed only for opaque-token architectures or fine-grained revocation. Defer.
- **Token revocation endpoint (RFC 7009)** — clients currently cannot self-revoke; admin revokes via `/admin/clients`. Add when a real client demands it.
- **Registration management (RFC 7592)** — `registration_access_token` + `GET/PUT/DELETE /oauth/register/:client_id`. Useful but not MVP-critical; v3.1 if planner excludes.
- **Dynamic refresh-token lifetime per client** (some clients want 7d, others 90d) — static 30d is fine for MVP.
