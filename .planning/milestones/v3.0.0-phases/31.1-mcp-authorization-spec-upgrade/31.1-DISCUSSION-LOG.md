## AS topology + audience model

### Q: Authorization Server topology?

| Option | Description | Selected |
|--------|-------------|----------|
| Single AS in dashboard | Dashboard exposes all OAuth endpoints; services are pure RSs verifying via JWKS. (Recommended) | ✓ |
| Per-service AS (federated) | Each service runs its own OAuth endpoints; clients register per-service. |  |
| External AS (Keycloak/Auth0) | Bolt on external provider; dashboard becomes a resource server too. |  |

**User's choice:** Single AS in dashboard

### Q: Token audience model?

| Option | Description | Selected |
|--------|-------------|----------|
| RFC 8707 resource indicators — per-service audience | Client requests `resource=...`; token has `aud=[that URL]`; service validates. (Recommended) | ✓ |
| Audience-agnostic | `aud='luqen'`; all services accept any dashboard token. |  |
| Hybrid | User tokens have audience; service-credentials tokens don't. |  |

**User's choice:** RFC 8707 resource indicators — per-service audience

### Q: Agent-specific resource indicator — multi-service tokens for dashboard chat?

| Option | Description | Selected |
|--------|-------------|----------|
| Dashboard mints per-request service tokens | AgentService mints short-lived per-service JWT per tool invocation; ties into Phase 32 D-03. (Recommended) | ✓ |
| Single multi-audience token | AgentService requests one token with `aud=[compliance, branding, llm, dashboard]`. |  |
| Agent runs only against dashboard MCP | Lock Phase 32 MVP to dashboard tools only; cross-service waits for Phase 33. |  |

**User's choice:** Dashboard mints per-request service tokens

## Scope vocabulary + grant coexistence

### Q: Scope vocabulary for MCP access tokens?

| Option | Description | Selected |
|--------|-------------|----------|
| Reuse existing scope suffixes (read / write / admin.*) | Same vocabulary Phase 30.1 scope-filter rules consume. (Recommended) | ✓ |
| MCP-specific per-resource scopes (mcp.scan.read, mcp.agent.use) | New namespace; more granular per-tool. |  |
| Hybrid — existing scopes + new `mcp.agent.use` | Minimal addition for agent-usage gating. |  |

**User's choice:** Reuse existing scope suffixes

### Q: Existing `client_credentials` grant on compliance/branding/LLM — what happens?

| Option | Description | Selected |
|--------|-------------|----------|
| Keep as-is for internal service-to-service | Per-service `/oauth/token` untouched; Phase 31.1 adds new external-client auth to dashboard. (Recommended) | ✓ |
| Consolidate to dashboard AS | Remove per-service `/oauth/token`; dashboard mints all tokens. |  |
| Deprecate client_credentials entirely | Replace with trusted-JWT internal path; no HTTP round-trip. |  |

**User's choice:** Keep as-is for internal service-to-service

### Q: Which org does an external MCP client's user token bind to?

| Option | Description | Selected |
|--------|-------------|----------|
| Active org from dashboard session at authorization time | Token inherits active orgId from session; matches existing UX. (Recommended) | ✓ |
| User must explicitly select org on consent | Consent screen includes org picker. |  |
| Token is org-agnostic; client specifies orgId per tool call | Breaks Phase 29 D-13 invariant. |  |

**User's choice:** Active org from dashboard session at authorization time

### Q: Can one user have multiple active MCP clients authorized at once?

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — independent per client_id, no cap | Real-world MCP usage (Claude Desktop + Inspector + IDE plugin). (Recommended) | ✓ |
| Yes but cap at 5 per user | Soft cap; 6th registration prompts revoke. |  |
| One active client per user | Registering new revokes previous. |  |

**User's choice:** Yes — independent per client_id, no cap

## DCR policy + consent UX

### Q: Dynamic Client Registration (RFC 7591) policy?

| Option | Description | Selected |
|--------|-------------|----------|
| Open DCR + user consent gate | Anyone can register; user consent is the safety gate. Matches spec + Claude Desktop's auto-register behavior. (Recommended) | ✓ |
| Admin-gated DCR | Admin approves registrations; breaks 'Claude Desktop just works' UX. |  |
| Allowlist DCR (redirect_uri domain matching) | Middle-ground; admin maintains allowlist globally. |  |
| Disabled | No /oauth/register; fails MCPAUTH-03. |  |

**User's choice:** Open DCR + user consent gate

### Q: Consent screen behavior?

| Option | Description | Selected |
|--------|-------------|----------|
| Full consent on first connect per client | First-time consent per client_id; remembered for subsequent with same-or-narrower scopes. (Recommended) | ✓ |
| Full consent on every authorization | Consent screen every time; OAuth 2.1 recommends remembered. |  |
| Auto-consent if client DCR'd in same session | Skip consent on immediate post-DCR flow. |  |

**User's choice:** Full consent on first connect per client

### Q: Consent screen scope granularity?

| Option | Description | Selected |
|--------|-------------|----------|
| All-or-nothing per client | User clicks Allow/Deny for the full scope bundle. (Recommended) | ✓ |
| Per-scope checkboxes | User ticks/unticks individual scopes. |  |
| Tiered (read auto / write click / admin.system reauth) | Progressive friction by scope tier. |  |

**User's choice:** All-or-nothing per client

## Key rotation + token lifetimes

### Q: Signing key: static PEM or JWKS endpoint?

| Option | Description | Selected |
|--------|-------------|----------|
| JWKS endpoint + key rotation | Dashboard publishes /.well-known/jwks.json; services cache + auto-refresh. Supports rotation. (Recommended) | ✓ |
| Static PEM env var (current) | Keep DASHBOARD_JWT_PUBLIC_KEY; redeploy to rotate. |  |
| JWKS + static PEM fallback | Try JWKS first; fall back to env. Resilient but loses rotation. |  |

**User's choice:** JWKS endpoint + key rotation

### Q: Access token lifetime?

| Option | Description | Selected |
|--------|-------------|----------|
| 1 hour | Standard OAuth 2.1 default. (Recommended) | ✓ |
| 15 minutes | Tighter blast radius; more refresh overhead. |  |
| 24 hours | Fewer refreshes; wider leak window. |  |
| Configurable per-deployment | Env var with 1h default. |  |

**User's choice:** 1 hour

### Q: Refresh token rotation and lifetime?

| Option | Description | Selected |
|--------|-------------|----------|
| Rotate-on-use + reuse detection + 30d absolute | Every refresh returns new token; reuse triggers chain revoke. OAuth 2.1 best practice. (Recommended) | ✓ |
| Rotate-on-use, no reuse detection, 90d absolute | Simpler; loses theft-detection. |  |
| Long-lived (1 year), no rotation | Simplest for clients; worst for leak scenarios. |  |

**User's choice:** Rotate-on-use + reuse detection + 30d absolute

### Q: PKCE code-challenge-method enforcement?

| Option | Description | Selected |
|--------|-------------|----------|
| S256 only (reject plain) | 400 if plain; universally supported. OAuth 2.1 BCP. (Recommended) | ✓ |
| S256 + plain (spec-default) | RFC 7636 default; plain cryptographically useless. |  |
| PKCE only for public clients | Public get PKCE-required; confidential may skip. |  |

**User's choice:** S256 only (reject plain)
