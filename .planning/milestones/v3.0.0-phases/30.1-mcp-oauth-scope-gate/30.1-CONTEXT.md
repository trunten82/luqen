<domain>
## Phase Boundary

**What this phase delivers:** The scope claim on an OAuth client-credentials bearer token must gate the dashboard MCP tool list at runtime. Today it does not — a `scope=read` client surfaces destructive tools (`dashboard_scan_site`) because the permission filter runs on a not-a-real-user fallback that hands back the legacy global `user` role's permissions, never letting the scope branch engage.

**What this phase does NOT deliver:**
- Cookie-session user behaviour changes (those users continue to be filtered by their real effective permissions; scope is not an input for cookie sessions).
- New OAuth scopes beyond `read | write | admin` (the hierarchy in `packages/core/src/mcp/http-plugin.ts:73-77` stays as-is).
- Admin UI changes to `/admin/clients` (scope registration already works).
- Per-tool RBAC rework (tool metadata `requiredPermission` in `packages/dashboard/src/mcp/metadata.ts` + `packages/dashboard/src/mcp/tools/admin.ts` stays as-is).
- Re-testing Phase 30 Checks 1, 2, 3, 4, 6 — those are ticked. Phase 30.1 only re-runs 5a and 5b.

**Phase requirement ID:** MCPT-05 (re-opens Phase 30 SC#4 acceptance).

</domain>

<decisions>
## Implementation Decisions

### Root cause (locked — verified end-to-end in live walkthrough 2026-04-18)

Trace of the gap:

1. `packages/compliance/src/api/routes/oauth.ts:156-161` signs the client-credentials token with `{sub: clientId, scopes, orgId?}`. **No `role` claim.**
2. `packages/dashboard/src/mcp/middleware.ts:71` reads `payload.role ?? 'member'` → always `'member'` for OAuth.
3. `packages/dashboard/src/permissions.ts:88-93` — `resolveEffectivePermissions` — not the admin shortcut, delegates to `roleRepository.getEffectivePermissions(sub, orgId)`.
4. `packages/dashboard/src/db/sqlite/repositories/role-repository.ts:171-179` — `getUserPermissions` looks up `sub` (a clientId) in `dashboard_users`, misses, and **falls back to the legacy global `user` role's permissions** (~10 perms including `scans.create`, `reports.view`, `branding.view` after migration 035).
5. `packages/core/src/mcp/http-plugin.ts:137-141` — `ctx.permissions.size > 0` is TRUE → `filterToolsByPermissions` runs. `filterToolsByScope` is **never** invoked for OAuth clients.

### Fix approach — locked decision: Option (b) empty-set on unknown sub

Two candidates were surfaced in the design-gap write-up:
- (a) AND-combine permission filter AND scope filter in `http-plugin.ts:139-141`.
- (b) Return empty set from `getUserPermissions` when the sub is not a real `dashboard_users` row.

**Chosen:** (b). Rationale:
- **Correctness:** An OAuth client is not a user. Handing it a user role's permissions was always a coincidence-of-convenience; the permission set it sees should be derived from scope, not from a DB-row-missing fallback.
- **Minimal blast radius:** The failing branch is unreachable for cookie-session users (they always have a matching `dashboard_users` row, so the `userRow === undefined` branch does not fire for them). `middleware.test.ts` Tests 4, 5, 6 all synthesize a real userId that matches the test storage stub — they do not exercise the unknown-sub branch, so removing the fallback does not touch them.
- **Preserves scope filter as authoritative for OAuth:** Once `ctx.permissions.size === 0`, `http-plugin.ts:141` falls through to `filterToolsByScope(toolMetadata, ctx.scopes)`. The scope filter in `packages/core/src/mcp/tool-filter.ts:32-58` then becomes the single source of truth for OAuth clients.
- **Defense in depth retained:** Option (a) also works but introduces a new behaviour (intersection) that would need its own unit tests. Option (b) leaves the filter semantics unchanged — permissions XOR scope — and relies entirely on existing, already-tested code paths.

### Behavioural contract after fix

For OAuth client-credentials tokens (any grant where `sub` is not a row in `dashboard_users`):
- `scope=read` → **5 tools**: `dashboard_list_reports`, `dashboard_get_report`, `dashboard_query_issues`, `dashboard_list_brand_scores`, `dashboard_get_brand_score`. `dashboard_scan_site` is filtered out because `scans.create` is not `.manage` / `.delete` / `admin.system` / `admin.org` but the scope filter already treats it as read-tier — WAIT: cross-check `tool-filter.ts:45-55`. `scans.create` maps to **read-tier**, so a read-scope token DOES see scan_site under pure scope filter. Scope filter rule needs a lookup or the filter needs extending. **See open-question OQ-1 below.**
- `scope=write` → **6+ tools**: all data tools + admin.org/admin.system admin tools (list_orgs, get_org, update_org + 5 service_connection tools).
- `scope=admin` → all 19 tools.
- For admin tools, the scope filter rule at `tool-filter.ts:47-54` treats `admin.users` as read-tier (no match to `.manage` / `.delete` / `admin.system` / `admin.org`). `scope=read` would therefore expose the 4 `admin.users` tools. That is also wrong — an OAuth read-only client should not be able to CREATE users. **See open-question OQ-1.**

### OQ-1 (LOCKED during planning — planner must resolve)

The current `filterToolsByScope` rule (`tool-filter.ts:47-55`) was designed for a different mental model — it treats anything not ending `.manage` / `.delete` / `admin.system` / `admin.org` as read-tier. That lumps `scans.create`, `admin.users`, and `reports.view` all together, which is wrong.

**Resolution strategy for planner:** the right move is to change `filterToolsByScope` so that:
- **`read` scope** → only allows tools whose `requiredPermission` ends `.view` OR is null. Nothing that writes, creates, deletes, or mutates.
- **`write` scope** → allows `.view`, `.create`, `.update`, `.manage`, `admin.users`, `admin.org` (but not `admin.system`, which stays admin-only).
- **`admin` scope** → allows everything.

Alternative: introduce a per-tool `minScope: 'read'|'write'|'admin'` annotation in `ToolMetadata` and key the filter off that instead of trying to infer from permission suffixes.

The planner picks ONE of those and codifies it. The annotation approach is more explicit and harder to break, but requires touching 19 tool registrations. The suffix-rule rewrite is a single-file change.

### Acceptance criteria (from ROADMAP.md, elaborated)

1. `scope=read` OAuth client → `tools/list` returns **only read-tier tools** (view/list/get). No `dashboard_scan_site`. No `dashboard_create_user` etc.
2. A direct `tools/call` for `dashboard_scan_site` with a read-scope bearer → HTTP 403 (filter-denied). NOT 200.
3. `scope=write` OAuth client → `tools/list` returns read-tier + write-tier tools; `dashboard_scan_site` is present; invoking it triggers Claude Desktop's `destructiveHint: true` confirmation UI.
4. Cookie-session users continue to receive the correct permission set — `middleware.test.ts` Tests 4, 5, 6 stay green.
5. Phase 30 VERIFICATION.md Checks 5a and 5b re-run end-to-end against the patched code and are marked ✓ (same Claude Desktop walkthrough, with `luqen-mcp-read` and `luqen-mcp-write` clients).
