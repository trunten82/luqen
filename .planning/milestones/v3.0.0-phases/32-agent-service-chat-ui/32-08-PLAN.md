<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin form submit → POST /settings | User-controlled text crosses into DB via updateOrgAgentDisplayName |
| DB agent_display_name → system prompt interpolation (Plan 02) | The name is fed into an LLM system prompt — prompt injection surface |
| DB agent_display_name → chat drawer HTML render (Plan 06) | The name is rendered in HTML via Handlebars — XSS surface |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-08-01 | Elevation of Privilege | Prompt injection via malicious display name | mitigate | Zod regex rejects `<`, `>`, `http://`, `https://`, `//` — blocks typical injection vectors. Plan 02's system prompt interpolation `replaceAll('{agentDisplayName}', name)` treats name as literal. Plan 02 T-32-02-03 adds additional sanitisation at interpolation time |
| T-32-08-02 | Tampering | XSS via display name in chat drawer header | mitigate | Handlebars default-escapes `{{agentDisplayName}}`. Zod regex also blocks the obvious vectors. Double defence |
| T-32-08-03 | Elevation of Privilege | Cross-org write — admin.org of Org A sets Org B's display name | mitigate | Route tenant check `!isAdmin && currentOrgId !== id` returns 403. Test 12 regression asserts. Mirrors the branding-mode handler's exact pattern |
| T-32-08-04 | Spoofing | CSRF — external form submits display name change | mitigate | Fastify CSRF middleware validates _csrf header/field; Test 11 regression |
| T-32-08-05 | Denial of Service | Overly long input DOS | mitigate | maxlength=40 client-side + zod max(40) server-side. Reject at both layers |
| T-32-08-06 | Information Disclosure | Admin sets display name to leak another org's brand | accept | Within-org concern; org admin's editorial responsibility. Audit trail through existing change-history service covers the paper trail |
</threat_model>
