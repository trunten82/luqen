| # | Surface | Audience | Route / Integration Point |

## Spacing Scale

Inherited from `:root` in style.css. All surfaces reuse these tokens; no exceptions.

| Token | Value | Usage in Phase 32 |
|-------|-------|-------------------|
| `--space-xs` | 4px | Badge icon gap, inline padding inside chat bubbles, form-error gap |
| `--space-sm` | 8px | Compact element spacing — gap between messages, chat-drawer header button gap, dialog button group gap |
| `--space-md` | 16px | Default element spacing — chat drawer horizontal padding, input-to-send-button spacing, dialog body padding |
| `--space-lg` | 24px | Section padding — drawer header padding, dialog outer padding, form-group margin between locked sections in prompt editor |
| `--space-xl` | 32px | Layout gaps — drawer header/body separation on large screens, modal outer padding (matches existing `.modal`) |
| `--space-2xl` | 48px | Major section breaks — empty-state vertical padding in first-open drawer |

**Exceptions (mandatory per WCAG 2.5.8 / 2.2 Target Size):**

- Speech button, Stop-stream button, Chat submit button, and the floating entry button MUST be **≥ 44 × 44 px** touch target. Pattern is identical to `.menu-toggle` in `main.hbs` (`style="min-height: 44px"` on the rescore button is the precedent; Phase 32 sets both width and height explicitly via a `.chat-btn--icon { width: 44px; height: 44px; }` utility added to style.css).
- Dialog Approve/Cancel button min-width 120px (matches existing `.consent__actions .btn { min-width: 120px; }`).

**No other exceptions.**

## Color

Inherited from `:root` (light) and `[data-theme="dark"]` (dark). Every hex value below has a dark-mode twin already defined in style.css — the UI-SPEC does not introduce any new color variables.

### 60 / 30 / 10 Split

| Role | Light Value | Dark Value | Usage in Phase 32 |
|------|------------|-----------|-------------------|
| Dominant (60%) | `--bg-primary` = `#ffffff` (light) / `#1f2937` (dark) | — | Drawer surface, dialog surface, admin-tab panel background |
| Secondary (30%) | `--bg-secondary` = `#fafafa` (light) / `#111827` (dark) AND `--bg-tertiary` = `#f8f9fa` (light) / `#374151` (dark) | — | Assistant message bubble bg = `--bg-secondary`; tool message bubble bg = `--bg-tertiary`; locked-fence card bg = `--bg-tertiary` (matches existing `.card--muted`) |
| Accent (10%) | `--accent` = `#15803d` (light) / `#4ade80` (dark) | — | **Reserved for: primary CTA only** |
| Destructive | `--status-error` = `#b91c1c` (light) / `#f87171` (dark) | — | Approve button on destructive-confirm dialog; "Response interrupted" retry card border; rate-limit error card; form-error text on `agent_display_name` validation |

### Accent Reserved-For List (explicit — never "all interactive elements")

Accent (`--accent` / `--accent-hover`) appears ONLY on:

1. The floating entry button fill (closed state) — the single universally-present entry point to the agent.
2. The primary chat Submit button (`.btn--primary` on the send control).
3. The user-message bubble border-left accent stripe (`border-left: 3px solid var(--accent)`) — mirrors existing `.stat-card` left-border pattern; announces "your turn" visually.
4. The tab-underline when `/admin/llm` extensions are active (inherited from existing `.tab--active`).
5. Focus ring on all interactive controls (`--focus-outline = 3px solid var(--accent)`, already inherited).

**Accent never appears on:** assistant-message bubbles, tool-message bubbles, dialog body, drawer header background, form field backgrounds, badge pills (except the tool-use "Supports tools" chip which uses `--status-success-bg` / `--status-success` — NOT accent).

### Additional Semantic Colors (existing tokens — no new ones)

| Token | Use in Phase 32 |
|-------|-----------------|
| `--status-success` + `--status-success-bg` | "Supports tools" chip on Anthropic model rows; "Sent" status on delivered messages; "Approved" outcome on past confirm rows |
| `--status-warning` + `--status-warning-bg` | "pending_confirmation" badge on the paused tool message; "rate_limited" inline card |
| `--status-error` + `--status-error-bg` | Destructive button on confirm dialog (`.btn--danger`); "Response interrupted" card; validation error on display-name field |
| `--status-info` + `--status-info-bg` | "Global only — per-org override disabled" info pill on prompt editor; manifest-size indicator pill |
| `--accent-template` = `#7c3aed` | **NOT used** — reserved for template system elsewhere |

### Dark Mode

Every surface inherits dark-mode values automatically via `[data-theme="dark"]` overrides in style.css (lines 101–146). The chat drawer, dialog, and admin extensions do not need per-surface dark-mode blocks — they read the same `--bg-primary` / `--text-primary` / etc. Custom drawer-only tokens are NOT introduced.

Color-only carriers are banned per WCAG 1.4.1:

- **Error state:** red border + `⚠` icon prefix (existing `.form-error::before { content: "⚠" }` pattern).
- **Destructive button:** red fill + explicit label text "Delete user", "Revoke API key" — not just "Confirm".
- **Tool message collapsed state:** caret icon `▸` / `▾` from native `<details>` — native element carries state semantics.
- **Pending-confirmation badge:** amber fill + `⏸` icon + text "Awaiting confirmation".

Critical flow spec:

   - Populate `#agent-confirm-summary` with friendly template (rendered server-side into the partial OR lazy-fetched from `GET /agent/confirm/:messageId/preview`).

4. If user closes the tab without deciding: pending state persists in DB indefinitely (D-30, no auto-deny TTL in MVP).
