<objective>
Create the two new dashboard MCP tool modules — `tools/scan.ts` (dashboard_scan_page)
and `tools/fix.ts` (dashboard_generate_fix) — following `tools/compliance.ts` EXACTLY
(80-PATTERNS.md). These are self-contained new files; wiring into metadata.ts /
server.ts / registerMcpRoutes happens in Plan 03 so both tools register atomically
(the tool-metadata-drift count invariant must never see a partial state).

`dashboard_scan_page` (MCPFIX-01) reuses the existing `@luqen/core` scanner with the
same finding shape (code/type/message/selector/context/runner). `dashboard_generate_fix`
(MCPFIX-02/03/04) HTTP-calls the Plan-01-extended llm `POST /api/v1/generate-fix` via an
`llmAccess` callback (mirroring `complianceAccess`), then enriches the payload with a
`legalContext` block (MCPFIX-03, compliance service) and a conservative `disclaimer`
(D-10). Tools are draft-only and non-destructive (D-09).

Purpose: deliver the agent-facing scan + fix capability surface.
Output: scan.ts, fix.ts, and a module-level test file proving findings shape, the
enriched fix payload (wcagCriterion + diff + legalContext + disclaimer), graceful
legalContext degradation, and the never-apply / conservative-framing guarantees.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Coding agent -> dashboard MCP tool | Agent-supplied url/html/wcagCriterion/issueMessage/platform cross into scanner + llm prompt |
| dashboard fix tool -> llm service (HTTP) | Forwarded fix request with caller HTML + platform |
| dashboard fix tool -> compliance service (HTTP) | wcagCriterion + jurisdictions for legal enrichment |
| scan tool -> target URL fetch | Agent-supplied url could point at internal infrastructure |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-80-05 | Spoofing/EoP (SSRF) | dashboard_scan_page url input | mitigate | Reuse scan-service isPrivateHostname check; reject private/internal/loopback hosts with errorEnvelope BEFORE scanning (Task 1) |
| T-80-06 | Information Disclosure | cross-org data leak via tools | mitigate | resolveOrgId from getCurrentToolContext only; tools read no org-scoped dashboard tables; orgId never accepted from args (mirrors llm MCP classification) |
| T-80-07 | Tampering | prompt-injection via issueMessage/htmlContext into fix | accept | Output is a draft for human review (D-09/D-10); no auto-apply; injected text cannot mutate any system |
| T-80-08 | Repudiation/Integrity | claiming "compliant" output | mitigate | D-10 DRAFT_DISCLAIMER on every fix payload; grep gate forbids asserting language (Task 2 acceptance) |
| T-80-09 | Elevation of Privilege | tool callable without permission | mitigate | requiredPermission gating wired in Plan 03 (scans.create / issues.fix); module exports tool names so Plan 03 metadata + runtime guard enforce |
| T-80-SC | Tampering | npm installs | mitigate | No new dependency expected; if proposed, RESEARCH.md Package Legitimacy Audit + blocking checkpoint required |
</threat_model>
