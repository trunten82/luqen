    Create `packages/dashboard/src/services/rescore/rescore-service.ts` with the following design:

    **Constructor dependencies (injected):**
    - `scanRepository: ScanRepository` — to list completed scans and read jsonReport
    - `brandScoreRepository: BrandScoreRepository` — to check existing scores (getLatestForScan) and insert new ones
    - `progressRepository: RescoreProgressRepository` — to track batch progress
    - `brandingStorage` — to resolve guidelines per site (`getGuidelineForSite`)

    **Public methods:**

    1. `async startRescore(orgId: string): Promise<{ status: 'started' | 'already-running' | 'no-candidates'; candidateCount: number }>`
       - Check progressRepository.getByOrgId(orgId) — if status is 'running', return `{ status: 'already-running', candidateCount: 0 }` (D-09)
       - Delete any stale completed/failed progress row for the org
       - Query all completed scans for orgId via scanRepository.listScans({ orgId, status: 'completed' })
       - Count candidates: for each scan, check brandScoreRepository.getLatestForScan(scanId) — if null, it's a candidate
       - If candidateCount === 0, return `{ status: 'no-candidates', candidateCount: 0 }`
       - Create progress row via progressRepository.upsert with status='running', totalScans=candidateCount, processedScans=0
       - Return `{ status: 'started', candidateCount }`

    2. `async processNextBatch(orgId: string): Promise<RescoreProgress | null>`
       - Get progress via progressRepository.getByOrgId(orgId) — if null or status !== 'running', return null
       - Query completed scans for orgId, ordered by created_at ASC
       - Skip scans up to lastProcessedScanId (resume point)
       - For each scan in next batch (max 50):
         a. Check brandScoreRepository.getLatestForScan(scan.id) — if exists, increment skippedCount, continue (BRESCORE-02)
         b. Resolve guideline: use brandingStorage.getGuidelineForSite(scan.siteUrl, orgId)
         c. If guideline is null/deleted, increment warningCount, continue (BRESCORE-04)
         d. Parse scan.jsonReport, collect all issues across pages (same pattern as branding-retag.ts)
         e. Project guideline to BrandGuideline shape (same projection as branding-retag.ts lines 37-62)
         f. Call `calculateBrandScore(brandedIssues, projectedGuideline)` — EMBEDDED MODE, no orchestrator (BRESCORE-05)
         g. Call brandScoreRepository.insert(scoreResult, context) with mode: 'embedded'
         h. Increment scoredCount
         i. Update lastProcessedScanId = scan.id
       - After batch: update progress row with new counts
       - If processedScans >= totalScans, set status='completed'
       - Wrap outer loop in try/catch — on error, set status='failed', record error message
       - Return updated progress

    3. `async getProgress(orgId: string): Promise<RescoreProgress | null>`
       - Return progressRepository.getByOrgId(orgId)

    4. `async getCandidateCount(orgId: string): Promise<number>`
       - Query completed scans for orgId
       - Count those without a brand_scores row (getLatestForScan returns null)
       - Return count

    **Important implementation notes:**
    - The rescore does NOT go through BrandingOrchestrator — it calls `calculateBrandScore` directly (BRESCORE-05: always embedded)
    - The issue matching step from branding-retag.ts is NOT needed here — rescore scores using the issues already stored in jsonReport. It needs to extract BrandedIssue[] from the scan's existing brandMatch data on each issue.
    - For scans that have no brandMatch data yet (pre-branding scans), the service should run the branding matcher first. Follow the pattern from branding-retag.ts: import the branding matcher and match issues against the guideline before scoring.
    - Batch size is exactly 50 (BRESCORE-03)
    - All scan iteration uses scanRepository.listScans, not raw SQL
