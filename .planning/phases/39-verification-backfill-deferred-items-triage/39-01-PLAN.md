<format_reference>
The three v3.1.0 VERIFICATION.md files (35, 36, 37) above are the canonical format. Required structure:

Frontmatter:
```yaml
---
phase: <phase-slug>
verified: <ISO-8601 timestamp>
status: passed | human_needed | failed
score: <N/M> must-haves verified
overrides_applied: 0
requirements_coverage:
  - id: <REQ-ID>
    status: SATISFIED | PARTIAL | UNVERIFIED
human_verification: []   # only if status=human_needed
---
```

Body sections (in order):
1. Phase Goal (one line, lifted from v3.0.0-ROADMAP.md)
2. Verified / Status / Re-verification metadata
3. Goal Achievement -> Observable Truths table (per-SC: # / Truth / Status / Evidence)
4. Required Artifacts table (file path / expected / status / details)
5. Key Link Verification table
6. Behavioral Spot-Checks table (commands run + result)
7. Requirements Coverage table
8. Gaps Summary (or "None")
9. Verifier signature footer

Lightweight backfill mode (per 39-CONTEXT.md):
- "Behavioral Spot-Checks" rows can cite "tests in <file>: <N pass> per <SUMMARY-path>" rather than re-running.
- "Required Artifacts" rows can cite SUMMARY evidence pointers; we are not re-grepping unless an SC explicitly demands it.
- Status PARTIAL = some evidence exists but not full coverage; UNVERIFIED = no evidence found.
- Every PARTIAL/UNVERIFIED row MUST appear in the Gaps section.
</format_reference>
