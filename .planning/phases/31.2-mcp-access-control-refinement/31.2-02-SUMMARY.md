## partitionScopes simplification (before/after)

**Before (31.1):**
```typescript
function partitionScopes(
  requested: readonly string[],
  userPermissions: Set<string>,
): { granted: readonly string[]; skipped: readonly string[] } {
  const granted: string[] = [];
  const skipped: string[] = [];
  for (const scope of requested) {
    if (scope.startsWith('admin.')) {
      if (userPermissions.has(scope)) granted.push(scope);
      else skipped.push(scope);
    } else {
      granted.push(scope);
    }
  }
  return { granted, skipped };
}
```

**After (31.2 D-12):**
```typescript
function partitionScopes(
  requested: readonly string[],
): { granted: readonly string[]; skipped: readonly string[] } {
  return { granted: requested, skipped: [] };
}
```

The `userPermissions` parameter is gone entirely — admin.* scopes can no longer be requested at the OAuth layer (D-10 rejects them at step 4 with 400 invalid_scope), so there's nothing to partition against. The `skipped` field is kept at `[]` so the view contract stays byte-stable for the mcp.use-holding path.

```typescript

  sub: input.sub,
```

## First-consent backfill SQL + atomicity note

```sql
UPDATE oauth_clients_v2
   SET registered_by_user_id = ?
 WHERE client_id = ?
   AND registered_by_user_id IS NULL
```

The `AND registered_by_user_id IS NULL` guard is the atomicity mechanism — a concurrent second consenter's UPDATE affects 0 rows because the column is no longer NULL by the time the second statement executes. SQLite's implicit transaction around a single UPDATE prevents any observable interleaving. No application-level locking needed. Covered by `recordRegistrationUser` Tests 1–3 (repository) + Test 4 (integration via POST /consent Allow).
