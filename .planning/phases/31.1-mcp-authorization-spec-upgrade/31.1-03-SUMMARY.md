# Phase 31.1 Plan 03: JWKS Verifier Swap + RFC 9728 RS Metadata — Summary

**One-liner:** Every service's RS256 verifier now pulls keys from the dashboard's `/oauth/jwks.json` via `jose.createRemoteJWKSet`, enforces RFC 8707 `aud` matching, and publishes `/.well-known/oauth-protected-resource` (RFC 9728) — so external MCP clients can discover the authorization server and tokens minted for one service can't be replayed against another.

### D-27 env var migration path (chosen: option a)

**Path (a) — keep the legacy env var + emit deprecation warning.** Rationale: operators can roll the JWKS-based deploy forward without synchronizing an env-var state flip across all four services in lockstep. The warning fires exactly once per process (guarded by a module-level `deprecationWarned` flag in `packages/dashboard/src/mcp/verifier.ts`) and the PEM path still enforces the aud check — the only downside is forgoing key rotation on that code path.

Full removal (option b) is deferred to a future phase (Plan 04 candidate or after one successful production cycle with the warning).

### Caching behavior inherited from jose

`jose.createRemoteJWKSet(new URL(jwksUri))` returns a resolver that:

- Fetches the JWKS lazily on the first `jwtVerify` call
- Caches keys per `kid` for the default TTL (5 minutes)
- Refreshes automatically on a cache miss (new `kid` seen in a token header → re-fetch)
- Deduplicates concurrent fetches (if two verifies race during a refresh, only one HTTP round-trip happens)

This is exactly the shape Plan 02 recommended — zero custom HTTP code in Luqen.

### Key rotation propagation

When the dashboard rotates its signing key (Plan 04 will add the admin UI; the repo already supports it per Plan 01):

1. Dashboard mints a new RSA keypair, inserts a row into `oauth_signing_keys` with a fresh `kid`, sets `created_at=now`.
2. `/oauth/jwks.json` immediately publishes both the new key and the retiring key (Plan 01's `listPublishableKeys()` returns all rows where `removed_at IS NULL`).
3. The `DashboardSigner.currentKid` flips to the new `kid` on the next bootstrap (or explicit reload). New tokens carry the new `kid` in their header.
4. In-flight tokens (signed with the OLD `kid` before the rotation) still verify — services' `createRemoteJWKSet` resolver keeps the old JWK in its set and looks it up by `kid`. No 401s during the transition.
5. After the D-25 overlap window (30d + 1h past the last token-issuance timestamp for that `kid`), a scheduled sweep sets `removed_at` on the old row. Services refresh the JWKS cache (TTL or explicit kid-miss) and stop accepting tokens signed with the retired `kid`.

Net result: rotation is invisible to running clients as long as they refresh access tokens within the 30d window.

## Deviations from Plan

### 1. [Rule 3 — Blocking Issue] branding + llm http.test.ts needed DASHBOARD_JWKS_URL override

- **Found during:** Task 2 GREEN verification.
- **Issue:** Compliance's `tests/api/helpers.ts` already sets `process.env['DASHBOARD_JWKS_URL'] = ''` before the test app boots, which routes the scoped preHandler through the locally-signed `verifyToken`. Branding and llm have no equivalent helpers.ts — their `tests/mcp/http.test.ts` files boot the server directly. After the Task 2 wire-up landed, the branding and llm `http.test.ts` suites started trying to verify Bearer tokens against a nonexistent dashboard JWKS endpoint (`http://dashboard.luqen.local/oauth/jwks.json`), causing 3 failures per package.
- **Fix:** Added `process.env['DASHBOARD_JWKS_URL'] = ''` at the top of each `beforeAll` block in `packages/branding/tests/mcp/http.test.ts` and `packages/llm/tests/mcp/http.test.ts`. Also hardened the branding + llm `server.ts` bootstrap to FALL BACK to the local `verifyToken` when the env var is empty — mirrors the compliance path (which already did this) so the three services behave identically.
- **Commit:** `1039d6d` (rolled into Task 2 GREEN).

### 2. Env var naming: DASHBOARD_JWKS_URL vs DASHBOARD_JWKS_URI

- **Rationale:** The plan exemplar used `DASHBOARD_JWKS_URI` for the services. I shipped `DASHBOARD_JWKS_URL` because the rest of the codebase uses `*_URL` suffix (DASHBOARD_PUBLIC_URL, COMPLIANCE_PUBLIC_URL, BRANDING_PUBLIC_URL, LLM_PUBLIC_URL). The dashboard's own verifier keeps `DASHBOARD_JWKS_URI` (matches the plan exemplar exactly and `packages/dashboard/src/mcp/verifier.ts` uses the term `jwksUri` internally). If this inconsistency bites operators, Plan 04 can unify.
- **Impact:** Purely documentation/ergonomics — no functional change.

### 3. Cross-audience integration test placement

- **Rationale:** Plan Task 3 text suggested placing cross-service integration checks in the dashboard test tree since the dashboard is the minting side. But dashboard's `tsconfig.json` sets `rootDir: src` and the dashboard package does NOT depend on `@luqen/compliance` / `@luqen/branding` / `@luqen/llm` (those services consume the dashboard's JWKS, not vice versa). The natural import path `../../../../compliance/src/auth/oauth.js` crosses the rootDir boundary. Moved the integration test to `packages/compliance/tests/integration/cross-audience.test.ts` — it uses the compliance factory with three different `expectedAudience` values. Because all three service factories are byte-identical by design (and each service has its own 7-test JWKS verifier suite proving the per-service behavior), this is equivalent to running the matrix against all three.
- **Impact:** Test coverage identical; the file lives in compliance instead of dashboard.

### 4. Shared verifier factory NOT extracted (D-34 left to Claude)

- **Rationale:** D-34 explicitly said extraction into `@luqen/core/mcp/verifier.ts` was "Claude's discretion". The three service factories are 45 lines each and byte-identical except for the local `TokenPayload` type narrowing per package. Extracting would require a generic type parameter that the caller has to project through — more surface, less clarity. Kept the triple as three separate implementations; the cross-audience test validates that the shape is equivalent in behavior.
- **Impact:** Three near-duplicate files. Future refactor candidate if a fourth service needs the verifier.

## Cross-audience integration test — D-04 proof

Verified in `packages/compliance/tests/integration/cross-audience.test.ts` (4 tests, all passing):

1. **compliance verifier accepts a token minted for compliance.** Token aud = `[https://compliance.test/api/v1/mcp]`, verifier expectedAudience = same. `jose.jwtVerify` resolves; `payload.sub` === `user-integration`, `payload.scopes` contains `read`.
2. **branding-shaped verifier rejects the same token (aud=compliance, not branding).** Token aud = `[https://compliance.test/api/v1/mcp]`, verifier expectedAudience = `https://branding.test/api/v1/mcp`. `jose.jwtVerify` throws `JWTClaimValidationFailed`; `verify(token)` rejects.
3. **llm-shaped verifier rejects the same token (aud=compliance, not llm).** Same setup, rejects.
4. **multi-audience token covers all three services (D-05).** Token aud = `[compliance, branding, llm]`, all three verifiers accept. Proves that external clients which need multiple services can request a single token covering them all.

The integration test is the explicit proof artifact for threat T-31.1-03-01 (spoofing via audience substitution). It complements the per-service Task 2 tests (Test 2 in each package's `oauth-jwks-verifier.test.ts`) — the per-service tests prove each verifier alone rejects a wrong-aud token; this integration test proves that the SAME token is accepted and rejected depending on which service's verifier factory it's presented to.

- T-31.1-03-01 (audience substitution) — every verifier passes `audience: expectedAudience` to `jwtVerify`. Cross-audience integration test (4 tests) is the explicit proof.
- T-31.1-03-02 (algorithm confusion HS256/RS256) — every verifier passes `algorithms: ['RS256']`. Test 3 in each per-service JWKS verifier suite asserts HS256 rejection.
- T-31.1-03-03 (JWKS leaks private keys) — `exportJWK(publicKey)` only; private material never exposed. Plan 01's `listPublishableKeys` query returns only public PEM columns.
- T-31.1-03-04 (JWKS DoS via re-fetch) — jose's built-in TTL caching + kid-miss-driven refresh. Zero custom HTTP logic.
- T-31.1-03-05 (key compromise → forever-valid forged tokens) — D-25 rotation window via `oauth_signing_keys.retired_at` / `removed_at` (Plan 01 schema ships this; Plan 04 wires the sweep).
- T-31.1-03-06 (.well-known leaks internal URL) — accepted by design; the URL is the public MCP entry point that clients MUST know.
- T-31.1-03-07 (service bootstrap mixes up verifier paths) — explicit in each server.ts: `createTokenVerifier(publicKeyPem)` stays on /api/v1/oauth/token; `createJwksTokenVerifier(jwksUri, expectedAud)` is the NEW call, scoped to the MCP preHandler only. Acceptance criteria grep-verified.
