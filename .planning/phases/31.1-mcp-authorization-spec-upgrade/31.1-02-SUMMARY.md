# Phase 31.1 Plan 02: OAuth 2.1 Authorization Server Endpoints — Summary

**One-liner:** Dashboard becomes a spec-compliant OAuth 2.1 Authorization Server (Authorization Code + PKCE + refresh rotation + client_credentials + DCR + JWKS + RFC 8414 metadata), plus a WCAG 2.1 AA consent screen and a defense-in-depth redirect_uri re-validation on the consent POST.

## What shipped

### Route surface (dashboard becomes an OAuth 2.1 AS)

| Method | Path                                          | Purpose |
|--------|-----------------------------------------------|---------|
| GET    | `/.well-known/oauth-authorization-server`     | RFC 8414 metadata — issuer, endpoints, supported grants, PKCE methods, scopes, auth methods, `Cache-Control: public, max-age=3600` |
| GET    | `/.well-known/jwks.json`                      | Mirror of `/oauth/jwks.json` (RFC 7517 shape) |
| GET    | `/oauth/jwks.json`                            | RSA public JWKs (`{kid, kty:'RSA', alg:'RS256', use:'sig', n, e}`) for every `removed_at IS NULL` signing key; `Cache-Control: public, max-age=3600` |
| GET    | `/oauth/authorize`                            | Authorization Code + PKCE browser flow. Consent coverage auto-approves and fast-paths to `?code=…&state=…`; missing/widening consent renders `oauth-consent.hbs` |
| POST   | `/oauth/authorize/consent`                    | Allow/Deny submit. 6-step order-of-checks (CSRF → redirect_uri re-validate → scope → PKCE → admin.system gate → approved branch) |
| POST   | `/oauth/token`                                | Three grants — `authorization_code` (PKCE S256 verify + single-use code + refresh mint) / `refresh_token` (rotate-on-use + reuse-detection → chain revoke) / `client_credentials` (confidential only) |
| POST   | `/oauth/register`                             | RFC 7591 DCR — 10 req/hour/IP via `@fastify/rate-limit` `errorResponseBuilder` returning JSON 429 `{error:'too_many_requests', statusCode:429}` |

### Issuer URL resolution

The `iss` claim on every minted JWT and the `issuer` field in `/.well-known/oauth-authorization-server` both resolve through the single helper `getDashboardIssuer()` (exported from `oauth-signer.ts`):

1. `DASHBOARD_PUBLIC_URL` env var when set
2. `https://dashboard.luqen.local` as fallback

Plan 03 verifier consumes this same URL pattern when building the JWKS resolver (`${issuer}/oauth/jwks.json`).

### How PKCE S256 is verified

`verifyS256Challenge(verifier, storedChallenge)`:

1. Length check: rejects `verifier.length < 43 || > 128` (RFC 7636 §4.1).
2. `createHash('sha256').update(verifier).digest('base64url')` computes the challenge.
3. `timingSafeEqual(computed, storedChallenge)` — constant-time compare to avoid leaking the stored challenge via timing side-channel (added on top of the plan's equality check — see Deviations below).

The token handler calls `verifyS256Challenge(body.code_verifier, row.codeChallenge)` immediately after `findAndConsume` and before minting. Failure → `400 invalid_grant`.

### How refresh reuse detection surfaces in the token response

`handleRefresh` in `token.ts`:

1. Computes `presentedHash = sha256(body.refresh_token).hex`.
2. Generates a fresh `rawRefresh` + its hash.
3. Calls `storage.oauthRefresh.rotate(presentedHash, newHash)` — Plan 01's repo detects reuse atomically and DELETEs the entire chain.
4. If `result.kind !== 'success'` → `400 {error:'invalid_grant'}` (same response for `expired` / `reuse_detected` / `not_found` so the client cannot distinguish them — defense against oracle attacks).
5. Cross-client guard: `result.parent.clientId !== client.clientId` → also `400 invalid_grant` (T-31.1-02-10).
6. Plan 04 will emit `agent_audit_log` `tool_name='oauth.refresh_reuse_detected'` when `kind==='reuse_detected'` (TODO noted in-code).

## Deviations from Plan

### 1. [Rule 2 — Security Hardening] timing-safe PKCE comparison

- **Found during:** Task 1 implementation of `oauth-pkce.ts`.
- **Issue:** Plan Step 2 used string `===` which is constant-time-unsafe for length-equal strings. A stored `code_challenge` is a 43-char base64url SHA-256 digest that sits in the DB; a timing oracle on the compare could let an attacker guess challenges byte-by-byte.
- **Fix:** Use `timingSafeEqual(Buffer.from(computed), Buffer.from(storedChallenge))` after first checking `computed.length === storedChallenge.length`. The behavioural contract (return boolean) is unchanged — all four PKCE tests still pass.
- **Commit:** `d4249f0`.

### 2. [Rule 3 — Blocking Issue] CSRF hook phase

- **Found during:** Task 2 GREEN — three consent POST tests (8, 9, 11b) were returning 403 on paths that should have returned 302/400.
- **Issue:** Route-level `onRequest: server.csrfProtection` runs BEFORE `preValidation` parses the form body, so `req.body._csrf` is undefined when CSRF validation runs → every legitimate POST was rejected with `FST_CSRF_MISSING_SECRET`.
- **Fix:** Switched the route to `preHandler: server.csrfProtection` — matches the dashboard-wide pattern already in `server.ts:778-788`.
- **Commit:** `20f0881`.

### 3. [Rule 3 — Blocking Issue] Test harness cookie round-trip

- **Found during:** Same Task 2 GREEN run as deviation #2.
- **Issue:** `primeCsrf()` helper was reading `res.cookies[0].value` (the raw unencoded cookie value) but `@fastify/secure-session` needs the URL-encoded form to decode. This made the test look like the POST had a fresh session (no stored CSRF secret).
- **Fix:** Switched to `res.headers['set-cookie']` and reuse the `name=value` pair directly (pre-encoded). `primeCsrf` docstring updated.
- **Commit:** `20f0881`.

### 4. Test count grew from 39 → 46

- **Rationale:** PKCE length-bound coverage splits into 4 `it()` blocks (42/43/128/129) for clarity (plan expected 2); redirect_uri validation in `/oauth/register` splits into 3 (https / localhost / 127.0.0.1) instead of 1 consolidated test; and Test 20 (confidential client secret) got a follow-up variant for HTTP Basic auth so both body and header auth paths are covered.
- **Impact:** All the plan-specified Tests 1–20 + 11b are present with their numbered assertions. The extra `it()` blocks extend coverage without changing behaviour.

### 5. pa11y fixture rather than live-server scan

- **Rationale:** Plan Step 5 allowed either Option A (live dashboard + captured session cookie) or Option B (fixture HTML from the HBS). The latter is simpler, faster, deterministic, and reproducible offline — and the fixture faithfully exercises the same Handlebars-rendered markup that production serves. Option B was picked.
- **Impact:** None — pa11y sees the same DOM that the production consent render produces. Verification evidence preserved in this SUMMARY plus `scripts/render-consent-for-pa11y.mts` (checked in) and `/tmp/pa11y-consent/*.html` (regeneration-safe via `tsx`).

## Notes for Plan 03 (verifier swap integration)

- **Issuer URL:** Use `getDashboardIssuer()` from `packages/dashboard/src/auth/oauth-signer.ts`. Plan 03's `jose.createRemoteJWKSet(new URL(issuer + '/oauth/jwks.json'))` should read the same function.
- **kid consistency:** Every token is minted with `kid = signer.currentKid`. Plan 03's verifier, when it gets a `kid` header on a token, will find the matching JWK published at `/oauth/jwks.json` for as long as the key has `removed_at IS NULL` (Plan 01's `listPublishableKeys`). Overlap window — 30d post-retirement — is enforced by D-25.
- **Payload shape:** Tokens carry `{ sub, scopes, orgId, aud, iat, exp, iss, role? }`. The existing dashboard verifier (`packages/dashboard/src/mcp/verifier.ts`) already extracts `sub`, `scopes`, `orgId`, `role`. Plan 03 needs to **ADD** `aud` verification (must contain the service's own MCP URL per D-04) AND swap `importSPKI` for `createRemoteJWKSet`. The rest of the verifier stays intact.
- **Service-side RS's:** `packages/compliance/src/mcp/verifier.ts`, `packages/branding/src/mcp/verifier.ts`, `packages/llm/src/mcp/verifier.ts` all follow the same swap. Consider extracting a shared factory to `@luqen/core/mcp/verifier.ts` (D-34) — Claude's discretion.
