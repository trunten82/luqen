# Phase 31.1 Plan 01: Data Layer Foundation (oauth_* tables + repositories) — Summary

**One-liner:** Five SQLite migrations (049-053) plus five org-aware repository interfaces and their Sqlite implementations that together supply every storage contract the OAuth endpoints (Plan 02), verifier swap (Plan 03), and admin UI (Plan 04) will consume.

## What shipped

### Migrations (APPEND-only, idempotent)

| ID  | Name                        | Purpose |
|-----|-----------------------------|---------|
| 049 | `oauth-clients-v2`          | DCR'd clients (RFC 7591). Includes `client_secret_hash` (bcrypt), JSON `redirect_uris` + `grant_types`, `token_endpoint_auth_method` CHECK in ('none','client_secret_basic'), FK to `dashboard_users.id` ON DELETE SET NULL. |
| 050 | `oauth-authorization-codes` | Short-lived (60s) single-use codes. Enforces `code_challenge_method = 'S256'` via CHECK constraint. FK CASCADE from `oauth_clients_v2` + `dashboard_users`. |
| 051 | `oauth-refresh-tokens`      | Rotation-chain store: `chain_id`, `parent_id` self-FK, `rotated` 0/1 CHECK, `absolute_expires_at` (30d hard cap). Three indexes: token_hash, chain_id, absolute_expires_at. |
| 052 | `oauth-user-consents`       | Per-client remembered consent with `UNIQUE(user_id, client_id)`, JSON `scopes` + `resources`. FK CASCADE to `dashboard_users` and `oauth_clients_v2`. |
| 053 | `oauth-signing-keys`        | JWKS source: `kid` PK, `public_key_pem`, `encrypted_private_key_pem`, `algorithm` CHECK = 'RS256', `retired_at`/`removed_at` nullable. |

All migrations use `CREATE TABLE IF NOT EXISTS` + `CREATE INDEX IF NOT EXISTS` and are applied transactionally by `MigrationRunner.run()`. Idempotent re-runs are no-ops.

### Repository interfaces and implementations

| Interface                 | Implementation                   | Key methods |
|---------------------------|----------------------------------|-------------|
| `OauthClientRepository`     | `SqliteOauthClientRepository`     | `register()` (returns raw `clientSecret` ONCE for confidential clients), `findByClientId`, `verifyClientSecret` (bcrypt.compare), `listByUserId`, `listAll`, `revoke` |
| `OauthCodeRepository`       | `SqliteOauthCodeRepository`       | `createCode` (throws `code_challenge_method must be S256` for non-S256), `findAndConsume` (atomic SELECT+DELETE inside `db.transaction()`) |
| `OauthRefreshRepository`    | `SqliteOauthRefreshRepository`    | `mint` (new chain: `chain_id = id`), `rotate` returning `RotateResult` discriminated union (`success`/`reuse_detected`/`expired`/`not_found`), `findByTokenHash`, `revokeChain`, `cleanupExpired` |
| `OauthConsentRepository`    | `SqliteOauthConsentRepository`    | `recordConsent` (ON CONFLICT upsert), `checkCoverage` (set-difference scope + resource), `listByUser`, `revoke` |
| `OauthSigningKeyRepository` | `SqliteOauthSigningKeyRepository` | `insertKey`, `findByKid`, `listActiveKeys` (retired_at IS NULL), `listPublishableKeys` (removed_at IS NULL → feeds JWKS), `retireKey`, `listRemovable(cutoff)`, `markRemoved` |

All five are wired on `StorageAdapter` and instantiated in `SqliteStorageAdapter` as `storage.oauthClients`, `storage.oauthCodes`, `storage.oauthRefresh`, `storage.oauthConsents`, `storage.oauthSigningKeys`.

### Encryption approach for `private_key_pem`

Per D-26 + T-31.1-01-02: the repository stores **ciphertext only**. The calling layer (Plan 02's token-signer) must:

1. Encrypt with `plugins/crypto.ts#encryptSecret(pem, key)` (AES-256-GCM, per-installation salt) BEFORE `insertKey`.
2. Decrypt only at mint time inside the token-signer — never persist the plaintext.

This mirrors the `service_connections` pattern (v2.8.0). The repo is intentionally ignorant of the encryption so the key is rotated at a single chokepoint in Plan 02.

### Reuse-detection invariant (D-29 / T-31.1-01-04)

Locked in `SqliteOauthRefreshRepository.rotate()`:

1. Look up the row by presented `token_hash` inside `db.transaction(() => …)`.
2. If the row's `absolute_expires_at < now`, DELETE all rows WHERE `chain_id = row.chain_id` → return `{kind: 'expired'}`.
3. If the row's `rotated === 1`, **revoke the chain** (DELETE WHERE chain_id) → return `{kind: 'reuse_detected', revokedChainId}`.
4. Otherwise UPDATE the row SET rotated=1, INSERT a child with `parent_id = row.id`, `chain_id = row.chain_id`, `absolute_expires_at = row.absolute_expires_at` (inherited — 30d cap preserved across rotations) → return `{kind: 'success', child, parent}`.

Test coverage pins every branch:

- `tests/repositories/oauth-refresh-repository.test.ts` Group 2: success flips `parent.rotated=true`, chains child with same `chainId` + `absoluteExpiresAt`.
- Group 3: attacker replays rawA after legit A→B rotation ⇒ `kind: 'reuse_detected'`, `revokedChainId` matches chain, rawB lookup also returns null (chain-wide wipe).
- Group 4: expired token ⇒ `kind: 'expired'`, chain purged.
- Group 5: `cleanupExpired()` returns count, leaves live tokens intact.

## Deviations from Plan

### 1. [Rule 3 — Blocking Issue] Test files relocated to `tests/` (plural), not `test/` (singular)

- **Found during:** Task 1 pre-flight check.
- **Issue:** Plan's `files_modified` frontmatter listed `packages/dashboard/test/db/sqlite/repositories/...`. Dashboard's `vitest.config.ts` only scans `tests/**/*.test.ts` (plural, flat layout — matches existing `tests/repositories/role-repository.test.ts` and the two Phase 31 test files).
- **Fix:** Placed all six test files under `packages/dashboard/tests/repositories/*.test.ts` to match the existing harness convention. Without this, vitest simply would not have discovered the tests and Task N verification would have silently reported 0 failures.
- **Commits:** all six test commits (above).

### 2. [Rule 1 — Bug] `chainId` widening in mint()

- **Found during:** Final `tsc --noEmit` before summarising.
- **Issue:** `const chainId = id` inferred the `randomUUID()` template-literal return type `${string}-${string}-...`. The later `chainId = parent.chain_id` assignment (where `parent.chain_id: string`) failed with TS2322.
- **Fix:** Changed to `let chainId: string = id` (one-line explicit type annotation).
- **Commit:** `849024a` (rolled into the Task 3 GREEN commit).

### 3. Consent suite grew from 6 → 8 tests

- **Rationale:** Plan called for "covered / not-covered / missing-scope / missing-resource" variants. Split them into four separate `it(…)` blocks rather than one multi-assertion block for legibility; added the explicit "no consent row exists" case. Net: 8 tests.
- **Impact:** All assertions still align with the `ConsentCoverageResult` shape in the plan. No new behaviour.
