<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| application code → SQLite | All writes pass through repository methods; raw SQL only inside repos |
| share token → permalink resolver (Plan 04) | Token in URL is untrusted; org membership check happens in route, not in repo |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-37-01 | Tampering | markMessagesSuperseded | mitigate | Org-guarded WHERE clause prevents foreign-org turn supersede; bulk update only touches rows belonging to (conversationId AND org_id) |
| T-37-02 | Information disclosure | getShareLink | mitigate | Repo intentionally does NOT enforce org check on getShareLink — that gate lives in the route handler (Plan 04) where session user is known. Repo simply hides revoked rows. |
| T-37-03 | Spoofing | createShareLink | mitigate | Token uses crypto.randomBytes(16) → 128 bits, base64url; statistically infeasible to guess |
| T-37-04 | Repudiation | revokeShareLink | accept | Audit logging for share lifecycle deferred to Plan 03/04 route layer (where userId + orgId are in scope) |
| T-37-05 | Denial of service | listForConversation | accept | Worst case is a user creating many share links for one conversation; per-conversation count is naturally low. Rate-limit on the create route (Plan 04) addresses abuse. |
</threat_model>
