Bridge contract (NEW — create as Task 1):

```typescript
// packages/dashboard/src/mcp/openapi-bridge.ts
import type { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import type { FastifyInstance } from 'fastify';
import { zodToJsonSchema } from 'zod-to-json-schema';

export interface RegisteredToolSnapshot {
  readonly name: string;
  readonly title?: string;
  readonly description?: string;
  readonly inputJsonSchema: Record<string, unknown>;
  readonly outputJsonSchema?: Record<string, unknown>;
}

/**
 * Walk the registered tool table on an McpServer instance and produce
 * a list of plain-shape snapshots whose Zod inputs have been converted
 * to JSON Schema. Source of truth: tool registration calls in
 * dashboard/src/mcp/tools/{data,admin}.ts.
 */
export function snapshotRegisteredTools(server: McpServer): readonly RegisteredToolSnapshot[];

/**
 * Inject one virtual OpenAPI operation per tool into the Fastify swagger
 * spec via app.addSchema + a custom transformer. Uses Fastify's
 * `app.swagger({ transform })` hook so the snapshot generator sees them.
 *
 * Operations are mounted under POST /api/v1/mcp/tools/{toolName} (virtual —
 * the real handler is the JSON-RPC endpoint at POST /api/v1/mcp; these
 * paths exist purely for spec discoverability per D-03).
 */
export function registerMcpOpenApiOperations(
  app: FastifyInstance,
  tools: readonly RegisteredToolSnapshot[],
): void;
```

Implementation notes for the bridge:
- The MCP SDK's tool registration internals: tools are stored on `(server as any).server._registeredTools` as a `Map<string, { description, inputSchema (Zod), outputSchema (Zod), handler }>`. Access via the same private field already used elsewhere in dashboard (see dashboard/src/server.ts comment at line 1066). Capture once at server-create time; do not mutate.
- Use `zodToJsonSchema(zodShape, { target: 'openApi3' })` to produce a JSON Schema compatible with `@fastify/swagger`'s OpenAPI 3.1 output.
- `@fastify/swagger` exposes a `transform` option on `app.register(swagger, { transform: ... })` which we cannot easily change post-registration. Alternative path: use the swagger plugin's `transformObject` option, OR inject paths via a Fastify GET-only no-op route registered with `{ schema: { hide: false, tags: ['mcp-tool'], operationId: 'mcp.tool.{toolName}', body: <converted>, response: { 200: Type.Any() } } }` — register one such virtual route per tool. The SIMPLEST approach: register one Fastify route per tool at `/api/v1/mcp/tools/{toolName}` that returns 405 Method Not Allowed (telling callers to use the JSON-RPC `/api/v1/mcp` endpoint), and use that route purely for OpenAPI spec discovery. This produces one operation per tool in the spec without breaking the JSON-RPC dispatcher.
- The JSON-RPC entry route (`POST /api/v1/mcp`) keeps the generic body schema added by Plan 41-04 NOT — Plan 41-04 explicitly excludes /api/v1/mcp*. Plan 41-05 owns this: add a JSON-RPC body schema + `Type.Any()` response, tag `['mcp']`. Body shape: `Type.Object({ jsonrpc: Type.Literal('2.0'), method: Type.String(), id: Type.Optional(Type.Union([Type.String(), Type.Number()])), params: Type.Optional(Type.Any()) }, { additionalProperties: true })`.
