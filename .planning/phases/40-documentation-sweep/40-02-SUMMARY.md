## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Generator missed second permission in chained `requirePermission` calls**

- **Found during:** Task 3 — RED gate of coverage test, missing `issues.fix`.
- **Issue:** Initial extraction regex used a non-global inner regex, so a route with `preHandler: [requirePermission('repos.credentials'), requirePermission('issues.fix')]` (see `packages/dashboard/src/routes/fix-pr.ts:202`) had its second permission silently dropped.
- **Fix:** Switched the inner `permCallRe` to `/g` and iterated over every `requirePermission` call inside each route's option-block tail.
- **Files modified:** `scripts/generate-rbac-matrix.ts`
- **Commit:** `ff20071`

**2. [Rule 1 — Bug] Stamped commit hash defeated the CI drift gate**

- **Found during:** Final acceptance check — re-running the generator after a new commit produced a one-line diff (only the `Last generated from commit: …` line changed).
- **Issue:** Including `git rev-parse --short HEAD` in the rendered output meant every commit invalidated the matrix and would force a regen-and-commit on every PR, defeating the drift gate (which uses `git diff --exit-code`).
- **Fix:** Removed the commit-hash line and the `execFileSync` import. Freshness is guaranteed by the CI re-run + `git diff --exit-code`, not by stamping the artifact. Updated the rendering function with a comment explaining the choice.
- **Files modified:** `scripts/generate-rbac-matrix.ts`, `docs/reference/rbac-matrix.md`
- **Commit:** `5302a83`

### Plan Implementation Choice

The plan suggested booting each Fastify service in-process to read `app.printRoutes()`. We instead used **deterministic regex extraction directly from source files**. Trade-offs:

- ✓ No test DB / OAuth mock plumbing required.
- ✓ No service-boot side-effects (DB connections, listeners) in CI.
- ✓ Works with the worktree-isolated executor model (no per-package `npm install` required for the generator itself; only the test step needs deps).
- ✓ Deterministic and trivial to audit.
- ⚠ Extraction is regex-based; new permission decoration patterns (e.g. a future `withPermission(...)` helper) would need an updated regex. Mitigated by the coverage test, which would go RED on any missed surface.

The plan permits this — its acceptance criteria specify the **output shape**, not the extraction mechanism, and the script's leading comment block satisfies the Task 1 documentation requirement.
