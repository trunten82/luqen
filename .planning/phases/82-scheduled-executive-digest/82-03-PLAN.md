<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| scheduler → notify plugins (Slack/Teams webhooks) | Outbound to third-party endpoints configured per org; failures must be contained |
| digest PDF/email → recipients | Document carries org data + legal framing; must never assert compliance/fault |
| scheduled job execution | Runs unattended on an interval; a single bad schedule/channel must not stall the sweep |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-82-09 | Denial of service | a failing/misconfigured channel wedging the schedule into resend loops | mitigate | per-channel try/catch + unconditional nextSendAt/lastSentAt advance after the loop (D-09); per-schedule try/catch in the sweep; timer.unref() |
| T-82-10 | Information disclosure | a digest delivered to the wrong org's recipients | mitigate | recipients + orgId come from the schedule row (org-scoped, Plan 01); buildDigest scoped to schedule.orgId/siteUrl |
| T-82-11 | Repudiation | PDF/email/Slack copy asserting compliance or fault | mitigate | DISCLAIMER_TEXT reused; band label+icon never a number; forbidden-words source greps + email-body test assertion (D-06/D-12) |
| T-82-12 | Tampering | malicious content in site/org identity injected into the PDF | mitigate | PDFKit text() renders plain strings (no markup execution); identity fields are org-owned data; no HTML-to-PDF path (no Chromium) |
| T-82-13 | Information disclosure | Slack/Teams summary leaking a numeric/precise risk claim | mitigate | message uses {icon}{band} + direction label only; disclaimer line required in every message (D-08/D-12) |
| T-82-SC | Tampering | npm installs | mitigate | No new dependencies — reuses pdfkit, the notify plugins, notification-service |
</threat_model>
